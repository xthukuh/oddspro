## Resolved issues (do not re-learn these)

1. **`_batch` race (src/utils.js):** originally resolved when the queue emptied while up to N tasks were still in flight — callers got partial arrays ("Found 37" vs 47 saved; old x-*.json dumps may contain holes/nulls). Fixed: resolve only when queue empty AND pending === 0.
2. **`_batch` crash bug:** catch handler called bare `warn()` → ReferenceError on any batch failure. Fixed to `console.warn` during extraction to utils.js.
3. **InnoDB gap-lock deadlocks:** parallel delete+insert transactions on the same table (standings, fixture stats) deadlock on unique-index gap locks. Rule: DB-writing `_batch` workloads run at concurrency 1.
4. **Timezone skew:** datetimes are stored as EAT wall-clock; MySQL server runs UTC. Session `time_zone` pinned to +03:00 via knex pool `afterCreate` (knexfile.js) so `NOW()` comparisons are correct. Fixtures fetched with `timezone=Africa/Nairobi`.
5. **zod vs live API:** api-sports fields go null unexpectedly (`league.round`). Keep response schemas tolerant: `.nullable().optional()` on anything not structurally essential.
6. **Correlation scoring lessons:** competition-name similarity must be a bonus, never a veto (identical team names were rejected at 0.816 when weighted 0.2); initialisms (BFA), token reordering (Flora Tallinn), reserve markers (II ≡ B), and club-type prefixes (JK/NK/US/AS/CS/CR/RS) all need explicit handling. Scorer = best-of(bigram dice, token-set dice, 0.9×overlap coefficient, initialism).
7. **`_date()` Date-instance bug (src/utils.js):** a valid `Date` argument short-circuited the single boolean chain and fell through to `new Date()` — every value became "now". Latent until Phase 6 because all earlier callers passed strings; mysql2 returns DATETIME columns as `Date` objects. Fixed 2026-07-02 with a dedicated ternary arm for valid Dates.
8. **Market identity:** map provider odds to canonical columns by `type_name`, never `type_id` — betika reuses `type_id` 19 across different team-total markets ("Z.PSV TOTAL" vs "ZWC.BELGIUM TOTAL"). Verified spellings live in `src/markets.js`.
9. **Pre-match bookmaker scores are garbage:** BetPawa reports 0-0 and Betika null for upcoming games — visualization/export only surface score/goals when the fixture status is final (`FT/AET/PEN/AWD/WO`).
10. **`Number(null) === 0` coercion trap:** a `param = null` default made the start pipeline sweep 1 date instead of 4 — `Number(null)` is `0` (valid integer), so the "invalid → default" fallback never fired. Rule: strict-parse optional CLI/config args with `parseInt(v, 10)` (null/undefined → NaN). Caught only by a live run; `node --check` and review both passed.
11. **Standings placeholder rows:** live `/standings` data can contain rows with `team.id = null` (TBD playoff/bracket slots) — crashed zod at 60/71 leagues once the 4-day sweep widened league coverage. Skipped pre-parse (no FK target to store). Same family as lesson 5: live data keeps finding new nulls.
12. **mysql2 NaN serialization:** binding `NaN` produces a bare `NaN` token in SQL → `Unknown column 'NaN'`. Any user-supplied value coerced with `Number()` must be validated (`Number.isNaN` → throw TypeError → 400) before it reaches knex. Found via the web tip filter (2026-07-06).
13. **In-response duplicates trip unique keys:** API-Football `/standings` can emit the same team twice under an identically-labelled group (MLS Next Pro league 909) — delete+insert doesn't protect against duplicates *within* the new batch. Rule: bulk-insert accumulators must be keyed on the DB's unique tuple (Map, not array), like the `leagues`/`teams` upserts already were. Fixed 2026-07-06 (`9dd388d`).
14. **`cmd | tee log` masks exit codes:** the first pipeline verification "passed" (exit 0) while node had actually crashed — the pipeline exit is tee's. Verify long runs by reading the output tail, or use `set -o pipefail`.
13. **LLM adjudicators over-veto without named criteria:** the first hot-pick AI prompt ("veto whenever fragile") produced 37/37 vetoes citing non-reasons (thin H2H = neutral by design, "value near threshold"). Fixed by stating that all quantitative gates already passed, enumerating concrete red-flag examples, explicitly listing what NOT to veto for, and feeding full aggregates (gf/ga/btts) instead of bare gate values → 16 confirmed / 21 vetoed with substantive reasons. Prompt lives in `src/ai.js`.
14. **LLM verdicts fabricate team news and argue backwards (2026-07-04, tip reviewer):** without web access the model asserted "key players injured" for obscure clubs (pure fabrication — mentioning web access in the prompt invites it), and even web-grounded it vetoed unders BECAUSE a team scores less (a factor that SUPPORTS the tip). Fixes in `src/ai.js`: "NEVER assert team news you have not verified" + an explicit direction check naming the tipped market ("your reason must argue AGAINST the tipped outcome") → 4/16 → 9/11 confirm/veto with directionally coherent reasons. Residual bad vetoes are measurable by design: vetoes flag but never clear a tip, so `performance`'s ai_impact.saved settles whether they help (consider a stronger `HOTPICK_AI_MODEL` if it goes negative).
15. **First settled losses were contextual, not model-internal (2026-07-05):** 282 settled tips read 74.5% hit-rate yet -5.6% ROI, and U hits vs misses were statistically identical on every in-model feature — the drivers were OUTSIDE the blend: `Friendlies Clubs` (-10.63u of -15.60u; preseason exhibitions between mismatched tiers invalidate rolling form), youth/reserve leagues (-23.3% ROI) and near-Unders U 2.5/3.5 (-21.8% ROI everywhere). Fixes: `TIP_CONTEXT_EXCLUDE` league gate in `tipEligibility` + `minUnderLine` 4.5 in `bestTip`. Method lesson: the persisted `tip_breakdown` cohort made honest counterfactuals cheap — and killed two seductive theories with data (stats-saturation ≥0.95 was fine; edge ≥ 0 gating REJECTED, edge-positive tips ran -6.0% ROI). Slice settled picks by league/market/side before touching model internals. Friendlies HELP over-2.5 hot picks (5/5) — the gate is tips-only.
16. **Anchored AI reviewers rubber-stamp; contradiction-only vetoes kill winners (2026-07-05, prompt v2):** v1 prompts said "they ALL passed… most deserve confirmation; expected veto rate: low" and fed the reviewer our own blended confidence — confirms became boilerplate and every settled hot veto was the same market-vs-stats numeric contradiction (re-litigating the rule engine): 4 winners killed vs 1 loser, -0.9u. The one failure class only an LLM could catch (friendlies context, literally printed in its prompt) went unflagged because nothing asked it to research context. v2 (`#p2` in `aiModelTag()` — prompt-version bump re-adjudicates upcoming rows automatically): research context + team news first, state an INDEPENDENT probability, veto mechanically only below break-even − 0.05 or on a verified disqualifier; contradiction alone is explicitly not a veto ground. Structured verdicts (probability/checks/citations) persist to `ai_review`/`tip_ai_review`; parsing is pure in `src/ai-parse.js`.
17. **Web "500 Internal Server Error" = the :3001 API is down, not a code bug (2026-07-08):** the dev proxy (vite :5173) and prod (Passenger) both surface a bare 500 when `src/server.js` isn't running or can't reach MySQL. First check: is `npm run serve` up and is the DB reachable? Restart the API after a DB blip (a killed/blipped MySQL leaves the knex pool with dead connections). The banner is transient by design now (auto-dismiss 3s + × button in `App.jsx`).
18. **Unstable effect deps → infinite refetch loop on server failure (2026-07-08):** `App.jsx` had `const providers = catalog?.providers ?? []` — a FRESH `[]` every render whenever the catalog fetch failed (null catalog). That new array fed the `selectedProviders` memo and the records-effect dep array, so each `setError` re-render changed the deps → refetch → fail → `setError` → … an infinite loop that hammered the failing server (the "why does the table keep refreshing on failure" perf bug). Fix: a module-level `EMPTY_PROVIDERS` constant as the stable fallback. **Rule: every dependency of a data-fetching `useEffect` must be reference-stable (primitives, or arrays memoized on stable inputs) — a `?? []`/`?? {}` fallback in a dep chain is a latent render loop that only bites when the happy path (which stabilizes the value via real data) doesn't run.**
19. **Aesthetics-only reskin stays green by construction (2026-07-09, Phase 23 iOS redesign):** the iPadOS reskin changed ONLY `web/` presentation and never touched a shared pure module (`magic-rules`/`ordering`/`filterValues`/`sortValues`) or any `src/db` logic, so the offline `node:test` suite (185) needed zero edits and stayed green through every commit — the cheapest possible proof of "no functional regression." Method that made it tractable: a Tailwind 4 `@theme inline` + CSS-var token layer flips light/dark from ONE place (`prefers-color-scheme` + a `:root[data-theme]` override), so restyling the dense 585-line table was `replace_all` utility swaps (`bg-white→bg-surface`, `text-slate-*→text-label*`, `sky→accent`, `emerald/rose→hit/miss`), NOT scattering `dark:` variants. Two reskin gotchas: (a) tests can't see layout — **verify a visual redesign by DRIVING it** (DevTools, both themes, iPad + phone widths); (b) an unreferenced new component file isn't bundled, so `build:web` passes even with a syntax error inside it until something imports it — the *importing* task is the real compile check. Rasterizing the `[OP]` app icon: `sharp` renders SVG `<text>` in system fonts fine on Windows (Arial present), installed as a throwaway in the scratchpad (never a project dep).
20. **Absolute dropdowns get clipped by an ancestor's `overflow` — anchor them `position:fixed` (2026-07-10, Phase 24):** the filter modal's value-picker dropdown (and any `MultiSelect` deep in a scrolling modal) was cut off because an `overflow-y-auto`/`overflow-hidden` ancestor clips absolutely-positioned children. Fix without portals: render the panel `position:fixed` with coords measured from the trigger's `getBoundingClientRect()` (flip up when space below is tight; reposition on `scroll`-capture + `resize`); a `fixed` element's containing block is the viewport (or the nearest `backdrop-filter`/`transform` ancestor — the full-screen scrim here), so intermediate `overflow` no longer clips it. Keep the panel a DOM child of the trigger wrapper so the existing outside-click `contains()` check still works. **Also (Phase 24): the filter builder's field labels must match the TABLE column titles** — reuse the same client label source (`BASE_COLUMNS` + catalog stat labels + market keys via `labelFor`) rather than emitting raw base keys, and derive value-picker options + which columns to even offer from the LOADED day's rows (`availableColumnKeys`/`distinctValues`) so the controls honestly reflect the date. `league` must filter client-side (its "Country - Name" display ≠ the `l.name` SQL column).

21. **The warehouse's past is self-healing — audit before "repairing" it (2026-07-10):** a request to "reset and rerun + repair past records" needed almost nothing on the past — a read-only audit (`diagnose.mjs`: unsettled-with-final-score, uncompleted finals, finals >48h missing stats) found ZERO backlog, because freeze-at-kickoff + settle-once plus the in-process auto-refresh keep settlement/completion/stats current. The forward `npm run start` sweep only refreshes ongoing/upcoming (both `updatePrematchSnapshots` and `updateHotPicks` select `kickoff > NOW()`); frozen past prematch/tip/hotpick SELECTIONS must never be recomputed — they'd use post-kickoff data and corrupt the honest scoreboard + backtest baselines. Measure the effect of new rules on history via the replay scripts (`backtest-hotpicks.js`/`analyze-safe-tips.js`), not by rewriting the ledger. **Ops gotcha:** `npm run serve` runs the scheduler in-process, so a manual CLI `npm run start` can deadlock with a light pass on the SAME odds rows (delete+insert gap locks) — stop serve for the sweep, then restart it.

22. **Manual-refresh "Insert SQL error" = an InnoDB deadlock from a SECOND concurrent writer (2026-07-11, Phase H):** the refresh button dumped a raw ~2 KB SQL error because the `teams` upsert inside `apisports._saveFixtureItems` deadlocked against another PROCESS writing the same rows — a second `serve`, a manual CLI sweep, or a backup cron overlapping the in-process light pass (the single-slot job guard only serializes writers WITHIN one process; it can't see other processes). Fix: pure `src/db/retry-rules.js` (`isRetryableDbError`/`retryDelayMs`/`withRetry`) wraps the idempotent refresh writers (`_saveFixtureItems` fixtures/teams/leagues + `store.saveMatches` per-match odds txn; `counts` moved OUTSIDE the txn so a retry can't double-count), and the UI shows "please try again" instead of the SQL. **Rule: run exactly ONE `serve` — the retry self-heals a rare race, but two always-on writers on the same rows keep fighting.** Same family as resolved issue #3 (gap-lock deadlocks). Live-verified on a restarted backend 2026-07-12: a full manual refresh completed with zero retries/deadlocks.
23. **A `backdrop-filter` (or `transform`/`filter`) ancestor traps `position:fixed` descendants to ITSELF (2026-07-11, Phase J):** the header popups' `fixed inset-0` tap-away backdrops only covered the nav strip — so a tap on the footer/table never dismissed them on mobile — because the nav's `backdrop-filter` makes IT, not the viewport, the containing block for any `position:fixed` child. Same CSS rule that resolved issue #20 relies on (a `fixed` panel escapes ancestor `overflow`). Fix: drop the backdrop and dismiss via a document-level `pointerdown` listener (`web/src/useOutsideDismiss.js`) that is independent of stacking/containing-block. **Rule: never rely on a `fixed inset-0` overlay for full-viewport coverage when any ancestor has `backdrop-filter`/`filter`/`transform` — use a document listener or portal to `<body>`.**
