**Must-fix — security / broken-in-prod:** ✅ ALL FOUR DONE 2026-07-15 (suite 426→435; live E2E on a
:3009 throwaway server, 24/24 — pow-gated admin bearer, live bot-filter toggle, the change-phone flood
429 repro, backoff growth, and the full must_change_pin block/verify/clear cycle).
- ✅ **H1 SMS-flood via change-phone.** `POST /api/auth/change-phone` (`src/server.js:235`) has NO `rateLimit()`
  (only signup/login do), and `issueOtp` (`src/auth.js:154-181`) skips the resend cooldown whenever
  `existing.phone !== phone` AND never writes `resend_count` on the rotate-update (170-173), so the `60·n`
  backoff and `OTP_MAX_RESENDS` cap can never trigger. With `SMS_ENABLED=1`, an authed unverified user
  alternating two E.164 numbers sends unlimited paid Bonga SMS to arbitrary numbers. Fix: rate-limit
  change-phone (DB-authoritative, not the spoofable in-memory IP limiter), gate the cooldown on user not
  phone, and increment `resend_count` on every send. (Confirmed by 3 independent finders + direct read.)
  *Fixed:* pure `otpIssueDecision` (sms-rules, 6 tests) gates issue on the USER's active row (reuse only
  same-phone+valid; reject 429 otherwise incl. the `max` cap); `issueOtp` bumps `resend_count` on every
  rotate-send; `changePhone` pre-checks the gate BEFORE mutating `users.phone`; `verifyOtp` rejects a
  row-phone ≠ user-phone code (`phone_changed` 409 - kills the raced-change verify bypass); `resendOtp`
  re-anchors to `user.phone`; the route adds a session-keyed (unspoofable) in-memory burst limit 5/15min.
- ✅ **H2 Human-pow gate blocks the admin settings API.** The `/api` human-gate (`src/server.js:115-123`)
  exempts `/challenge`, `/human`, `/visits/summary` and API_TOKEN bearers — but NOT `/api/admin/settings*`,
  which `requireAdminDual` promises to serve to ADMIN_TOKEN clients. In the documented prod config
  (`HUMAN_POW_ENABLED=1`, `ADMIN_TOKEN` set, `API_TOKEN` unset) every admin-settings call 401s before its own
  auth runs. Fix at altitude: make the gate recognize route-owned bearer auth generically (or mount
  self-authenticating routers before the gate) instead of growing the per-path allow-list. (3 finders + read.)
  *Fixed:* pure `bearerMatches(header, secrets)` (human-pow.js, constant-time, skips unset secrets, 2 tests);
  ONE `MACHINE_BEARERS = [API_TOKEN, ADMIN_TOKEN]` list honored by BOTH blanket /api gates (the API_TOKEN
  wall gains ADMIN_TOKEN recognition too - same bug class); the `/visits/summary` path exemption removed;
  `requireAdmin`/`requireAdminDual` reuse the helper (drops the duplicated local `safeEqual`, part of C1).
- ✅ **H3 Dead `features` overrides.** `SMS_ENABLED` / `BOT_UA_FILTER_ENABLED` / `GEO_RESOLVE_ENABLED`
  (`src/db/settings-rules.js:33-35`) are admin-editable but NO consumer reads them via `settings.effective()`
  — `sms/index.js:41`, `server.js:45`, `geo.js:117` read `config.*` directly (the bot-UA middleware even
  registers at module-eval, before `loadOverrides()`). An admin toggles them, gets `restart_required`,
  restarts — nothing changes, while `GET /api/admin/settings` reports `effective:true`. Fix: route these
  consumers through `effective()`, OR drop the three entries until a consumer late-reads them. (5 finders.)
  *Fixed:* `smsEnabled()` late-reads `effective('SMS_ENABLED')` (→ live:true); the bot-UA middleware is
  always registered and late-reads the flag per request (→ live:true); `startGeoScheduler` reads
  `effective('GEO_RESOLVE_ENABLED')` (boot runs `loadOverrides()` first, so restart honors the override —
  stays live:false, the restart_required promise is now honest). Catalog live flags + tests updated.
- ✅ **H4 `must_change_pin` not enforced.** The migration flags the seeded admin `must_change_pin=1` and
  `auth.js` clears it on PIN change, but no middleware enforces it — a default-PIN (`0000`) admin session
  works on every route. Fix: in `authGuard`, once the session resolves, if `must_change_pin` reject every
  request except `PUT /api/auth/profile` with 403 `{pin_change_required:true}`. (Background security review.)
  *Fixed:* pure `mustChangePinBlocks(method, path)` (auth-rules.js) enforced in `authGuard` AND
  `requireAdminDual`'s session branch (the flagged seeded admin must not drive the admin API either).
  Exempt set widened beyond the letter: `PUT profile` + `POST logout` (always allow leaving) + `GET me`
  (the SPA re-discovers the flag on reload) + `POST verify-otp`/`resend-otp` (PUT profile requires a
  VERIFIED phone, so blocking verify would dead-end an unverified flagged user — unreachable today, but
  the guard must not construct dead ends).
