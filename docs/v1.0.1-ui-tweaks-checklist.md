## R26 — Tip runners-up at a glance (added mid-session 2026-07-11)
> `bestTip` already persists runners-up in `tip_breakdown`; each runner-up ✓/✗ is settle-able client-side via `tipHit(market, hg, ag)` on the final score.
- ⬜ **R26a** Tip cell shows top-3 picks (chosen + 2 runners-up) stacked, each with hit/miss tick; chosen is bolder/slightly larger (it's the sort value), runners-up smaller — *B*
- ✅ **R26b** Filter tip runners-up via a value prefix on the `tip` column. Grammar: `[H|M]?\d?:value`.
  - `\d` = candidate index: `1:O 2.5` (chosen/default), `2:O 2.0` (2nd), `3:O 2.5` (3rd); bare value = 1st.
  - `H|M` = settled-outcome filter (added 2026-07-11): `H1:O 2.5` = 1st-pick O 2.5 that HIT, `M1:O 2.5` = that MISSED; no `H`/`M` = all outcomes. Combines with the index (`H2:...`, `M:...`).
  - Strip the prefix in value preprocessing before the op runs; resolve the Nth candidate market from `tip_breakdown` and grade hit/miss via `tipHit` on the final score — *C*
  - DONE (2026-07-11): pure `parseTipFilter`/`tipCandidateMarket`/`tipCandidateOutcome` (filterExpr.js), `evalCond` tip-prefix branch (AND-combines the outcome gate with the market predicate), prefixed-tip conditions forced client-side in `splitFilters`, FilterBuilder offers contains/not-contains on tip + prefix hint. Tests: `tests/tip-filter.test.js` (14) + splitFilters routing (1). Colon-required prefix = backward compatible (plain markets have no colon). Uniform tipHit-from-score grading = display parity with the tip-cell runners-up ticks.
- ⬜ **R26c** Footer status totals extended with 2nd + 3rd runner-up hit-rates — *E/footer*
- ⬜ **R26d** Betslip: per-leg tip-market select (1st default / 2nd / 3rd runner-up) that re-computes odds/prob/outcome — *D*
