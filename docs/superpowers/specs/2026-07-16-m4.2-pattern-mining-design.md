## 1. Why this spec exists at all — the premise changed

The backlog (`docs/emergence-patterns-m4-backlog.md`, 2026-07-16) proposed a
"Phase A: read-only pattern-mining scripts over settled history … + a bounded
automated config-search with OOS validation".

**That was already built and run on 2026-07-14** — `docs/precursor-patterns.md`:
203 enumerated candidate patterns over 16,518 leak-free fixtures, temporal-OOS
holdout (oldest 70% train / newest 30% test), day-clustered bootstrap CIs
(1,000 draws), and Benjamini–Hochberg FDR at q=0.10. 149 patterns survived
Tier A; **zero cleared break-even on their bettable (≥1.20) slice**.

The backlog was written two days later and did not account for it. So M4.2 is
**not** "build the mine" — the mine exists. M4.2 is:

1. **Rescue** the methodology (it lives in gitignored `tmp/`).
2. **Mine the surface the precursor mine never touched** — the *tip ledger*
   (`tip_breakdown`, runners-up, AI verdicts), as opposed to the *warehouse
   reconstruction* (raw team aggregates) it did touch.
3. **Report honestly**, and pre-register the gates that a later, separate
   M4.2b would use to ship anything.

### H5 is dropped, and this is a finding, not an omission **[CLAUDE'S CALL]**

The user's observation — bookmakers misprice glamour longshots (a 20x home win
that lands) — is real as an *anecdote* and inverted as a *statistic*. The ≥10x
band went **2-for-153 (1.3%) at an average price of 16.04 ⇒ ≈ −79% flat EV**:
the worst-priced region of the entire board, and a textbook **favorite-longshot
bias** (books shade longshots hardest precisely because punters chase the
memorable 20x winner). With **2 positive examples**, no discriminator can be
fit, let alone validated — a "spotter" here would be fitting noise to two
games and would ship a −79% EV surface with a 🔥-style badge on it.

**This does not refute the contrarian thesis — it corroborates it, aimed the
other way.** "Where public money concentrates, the price gets worse" is exactly
what −79% on the glamour bet looks like. H5's intuition is retained and
redirected into H7, where it is testable.

*Reopen condition:* revisit if the ≥10x band ever accrues ≥30 winners.

## 5. Method — the anti-false-positive controls (non-negotiable)

Every control below already has precedent in this repo. This is not new rigor;
it is the rigor that killed "X2 +15% EV" and the runner-up swap.

1. **Pre-registration.** The hypothesis registry is a *literal constant* in
   `mine-rules.js`, written and committed **before** the mine runs. A pattern
   discovered post-hoc is reported in a clearly-labelled EXPLORATORY section
   and is **never** eligible for an M4.2b ship gate.
2. **Temporal OOS.** Train = oldest 70% of *days*, test = newest 30%. Split on
   whole days, never rows — same-day tips are correlated.
3. **Day-clustered bootstrap.** Resample whole kickoff-days (1,000 draws) for
   cluster-robust CIs on precision and on lift. Per-day legs are correlated;
   row-level resampling would fabricate confidence.
4. **Benjamini–Hochberg FDR at q=0.10** over all pre-registered tests.
5. **Volume floor.** train ≥ 100 and test ≥ 40 tips, else "underpowered", not
   tested. (Lower than precursor's 200/80 — the tip ledger is 1,089 rows, not
   16,518. Recorded honestly as reduced power, not hidden.)
6. **Price test.** Every survivor is re-scored at its *real stored price* for
   flat-stake EV against break-even (1/price). **Price-blind precision is
   explicitly NOT a result.**

### The honesty rule this milestone exists to respect

> A pattern that raises hit-rate is a **booster** (it buys slip survival).
> A pattern that clears break-even at real prices is an **edge** (it buys
> profit). The warehouse has produced many of the first and **zero** of the
> second. The findings doc must label every result, and must never let a
> booster be read as an edge.

**The class vocabulary is closed and exhaustive.** `evaluatePattern` returns
exactly one of these, and both the script (§12) and the findings doc use the
same five words:

| Class | Meaning |
|---|---|
| `edge` | Survives every control **and** flat EV > 0 at real prices. None has ever been found. |
| `booster` | Survives every control, lift CI excludes zero, but flat EV ≤ 0. Buys survival, not profit. |
| `refuted` | Tested with adequate power; lift CI includes zero, or fails OOS/BH. |
| `underpowered` | Below the §5.5 volume floor. Not evidence of absence — reported with its n. |
| `unbettable` | Real lift, but the price lives below the 1.20 floor, so it cannot be acted on. (The precursor mine's AU 2.5 / U 4.5 trap — a distinct outcome from `refuted`, and conflating them would lose the most important lesson in `precursor-patterns.md`.) |

## 6. The pre-registered hypotheses

Committed before the run. Each states the population, the claim, and the metric.

**PR-1 — Cascade ladder (H1).** Population: settled tips whose market is
`O <line>`. Claim: the fixture clears `O 1.5` materially more often than the
tip itself lands. Metric: P(cleared O 1.5 | tipped O 2.5) vs P(tip hit).
*Brainstorm probe suggests 85.1% vs 70.3% (n=101) — treat as the hypothesis,
not the result; it must survive OOS + CI + price.*
Ship-relevance: a laddered leg only helps if O 1.5's **real price** at those
fixtures clears break-even at the laddered rate (≈1.175 at 85.1%). O 1.5 is a
live −5.4% loser (`precursor-patterns.md`), so **expect a sub-1.20 trap.**

**PR-2 — Runner-up configuration (H2), the flagship.** Population: 1,076
settled tips with ≥2 runners-up. Claim: the *configuration* — the ordered
multiset of runner-up markets — carries outcome signal beyond the winning tip's
own confidence. Pre-registered configs:
  - (a) **Straddle**: runners-up contain both `O k` and `U k` for the same line
    k (the user's `2:O 3.5`, `3:U 3.5` observation) ⇒ predicts high-variance /
    high-scoring.
  - (b) **Concord**: winner and both runners-up fall in the same market family,
    where family is taken from the existing `perf-rules.js` market-group
    labeller (the same grouping `computeCalibration` already buckets by — not a
    new taxonomy) ⇒ predicts higher tip hit rate.
  - (c) **Confidence gap**: winner.confidence − ru1.confidence ≥ 0.10 ⇒
    predicts higher tip hit rate.
Metric: tip hit rate + flat EV, per config, vs the unconditional 71.0% base.

**PR-3 — Miss commonality (H3).** Population: 316 settled misses vs 773 hits.
Claim: misses concentrate in identifiable cells (market family × price band ×
sample thinness × blend-parts). Metric: per-cell miss rate vs base 29.0%; a
cell is an "avoid candidate" only if its **EV** is worse than the book's
average, not merely its hit-rate.

**PR-4 — Consensus anti-signal (H7), the founding thesis.** Three independent
lenses, tested separately (each gets its own BH slot):
  - (a) **Price shortness**: does hit-rate-per-price-paid worsen as the tip
    gets shorter? *Probe: <1.30 → −7.3% EV vs 1.30–1.60 → −3.3%. Directionally
    pro-thesis.*
  - (b) **Market-vs-stats agreement**: `|market_prob − stats_prob|` from
    `tip_breakdown`. Claim: high agreement (the "consensus trap") underperforms
    its price; disagreement outperforms.
  - (c) **AI verdict**: confirm vs veto. **Pre-declared UNDERPOWERED at n=61**
    (75.0% vs 72.7%). Reported with its CI and explicitly excluded from any
    ship gate until n ≥ 300. It is registered *now* so that when the sample
    arrives the test is already pre-registered rather than fitted.

Lens (b) is the thesis's sharpest testable form and runs on all 1,076 rows.

## 7. M4.2b ship gates (pre-registered here; M4.2 ships no ranking change)

M4.2 is **read-only**. Nothing in it touches `bestTip`, `scoreTip`,
`safeQualifies`, or any ranking — same discipline as M4.1 (collection only,
"answered by replay, never by assertion").

A pattern earns an M4.2b ship **only if all four hold**:

1. It was **pre-registered** (§6). Exploratory finds do not qualify.
2. It survives **temporal OOS** (test precision ≥ train − 5pp) **and**
   **BH-FDR at q=0.10** **and** its day-clustered lift CI excludes zero.
3. It is validated on the **live ledger** via the existing LODO harness
   (`scripts/analyze-safe-tips.js` idiom), not only in the mine.
4. Its intended use matches its class: a **booster** may only gate/rank the
   Safe pool (survival); only an **edge** may ever be labelled +EV.

## 9. Testing

`tests/mine-rules.test.js` — offline `node:test`, no DB, no network, matching
the suite's existing character (564 passing).

- **Statistics correctness** (the part most likely to be silently wrong):
  - `benjaminiHochberg` against a hand-worked example with known rejections;
    q=0 rejects nothing; all-p=1 rejects nothing; ordering independence.
  - `dayClusteredBootstrap` resamples **days not rows** — asserted by feeding
    2 days × 50 identical rows and confirming the CI is *wide* (a row-level
    resampler would report a false-precise CI; this test is the whole point).
  - `temporalSplit` splits on day boundaries, never mid-day.
- **Feature extractors** (pure, table-driven):
  - `configSignature` on the two real breakdown shapes captured in the probe;
    null/malformed/absent `runners_up` → null, never throw (persisted JSON is
    external data — the project's zod-tolerance lesson applies).
  - `cascadeLadder` — `O 2.5` tip with 3 goals ⇒ cleared O1.5 ✓, O2.5 ✓,
    O3.5 ✗; boundary at exactly 2 goals for line 2.5.
  - `missProfile` / `consensusProxies` on rows with `api_prob: null`
    (2-part blends are the majority — the probe confirms `weights.api: null`).
- **The honesty invariant, encoded as a test:** `evaluatePattern` must return
  both `precision` and `flatEv`, and a pattern with precision > base but
  flatEv ≤ 0 must classify as `'booster'`, never `'edge'`. A sub-1.20 lift must
  classify `'unbettable'`, never `'booster'`. These are the rules the whole
  milestone exists to protect, so they get tests, not comments.

## 10. Error handling

- `tip_breakdown` is **external-ish data** (persisted JSON from an older code
  version): every extractor is total — malformed/absent input returns `null`,
  never throws. Prior art: `tipHitSafe`, the tolerant `apisports-events.js`.
  Pre-phase-14 rows lack `runners_up` entirely (13 of 1,089) and must simply
  drop out of PR-2's population, counted and reported.
- The script is **read-only**: no writes, no API calls, no AI calls. It cannot
  corrupt the frozen ledger by construction.
- Underpowered tests report `"underpowered (n=…)"` rather than a number. A
  suppressed test is still listed — silent omission reads as "not found".

## 12. Success criteria

1. `node scripts/mine-patterns.js` runs read-only and prints every
   pre-registered test with n, train/test precision, day-clustered lift CI,
   BH verdict, real-price flat EV, and one class from the closed §5 vocabulary
   (`edge` / `booster` / `refuted` / `underpowered` / `unbettable`).
2. `tests/mine-rules.test.js` passes offline; full suite stays green (564+).
3. `docs/emergence-patterns-findings.md` states, per hypothesis, what was
   claimed, what was measured, and the verdict — **including the refutations**.
   A milestone whose honest output is "H1, H2, H5 all failed" is a **success**;
   the failure mode is a mined false positive reaching live ranking.
4. The backlog doc is updated to point here, so no future session re-mines
   what `precursor-patterns.md` already settled.
