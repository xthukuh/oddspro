# Admin Dashboard Improvements — Program Spec

Stamp: `2026-07-19-0104` (spec/plan/checklist share it). Branch: `feature/admin-dashboard-improvements`.

## Problem

1. **Production visitor tracking is dead-on-arrival.** The cPanel deploy serves the SPA statically from `public_html`; HTML navigations never reach Express, so the `shouldLogVisit` middleware (`src/server.js:79`) never fires. Verified live: `GET /api/visits/daily-unique` works (table + code deployed) and returns `{unique:0,total:0}` forever.
2. **The admin surface is minimal** (two tabs: raw-input SettingsEditor + DataLab) while the operational need has grown: settings UX, user management, SMS broadcasting, DB tooling, engine-performance insight.
3. **OTP SMS sending fails** ("We couldn't send the code…") — postman-collection diff shows our send is `application/x-www-form-urlencoded` where the vendor documents `multipart/form-data`; and there is no fallback channel, so a failed SMS blocks onboarding entirely.
4. **No legal layer** (Privacy Policy / Terms) despite first-party behavior tracking and SMS messaging.
5. **`.env` sprawl**: ~35 behavior knobs live in env although a curated admin-settings override system exists.

## Goals

- First-party visitor + feature-usage analytics that work on BOTH topologies (dev Express-served, prod Apache-static), normalized to reduce duplication, with duration derived from activity — data accumulates for future behavior analysis. Non-intrusive, first-party only, disclosed in the Privacy Policy.
- A full admin area (own `#admin/<section>` sub-navigation, lazy chunk): Dashboard, Settings, Users, Messaging, Data Lab, Performance, Database, About.
- Settings: every non-secret knob admin-editable (label + hint + per-field reset, semantic-dirty sticky save); `.env` reduced to credentials/endpoints/boot-infra/VITE_*.
- Working SMS (multipart fix) + reusable templates (`[OP] ${message}`) + broadcast campaigns with filter- or selection-based audiences, opt-out honored.
- Email OTP fallback + admin manual verification + OTP-authorized critical changes — no user permanently blocked at OTP verification.
- Legal consent: Privacy Policy + Terms of Use ("Oddspro", info@oddspro.ke, Kenyan law incl. DPA 2019), required signup checkbox, versioned acceptance.
- DB tools: overview/health + chunked gzip export/import with progress for local⇄remote sync.
- Engine-performance visualizations that surface patterns hard to spot mathematically (honesty rules preserved: pre-binned, n-badged, no EV claims).
- Seamless, manageable deployment transitions (M14, added 2026-07-19 post-M4): admin-scheduled maintenance windows — advance warning banner on every client, hard 503 during the window, and clients that cache the schedule and switch in/out of maintenance mode autonomously so operations proceed undisturbed.

## Architecture notes

- **Pure-rules discipline**: every new subsystem ships a zero-import (or zod-only) rules module + offline node:test coverage: `track-rules`, `campaign-rules`, `admin-rules`, `transfer-rules`, `scorecard-rules`, extended `settings-rules`/`sms-rules`/`migrate-rules`. Services stay thin knex orchestration.
- **Guards**: new admin routes are `requireAdminRole` (session-only) + `csrfOk` on mutations; `/api/visits/summary` widens to `requireAdminDual`. Beacon routes are public + `csrfOk` + small body limits + best-effort (analytics may never break the app).
- **Beacon CSRF**: `fetch(..., {keepalive:true})` with `X-Requested-With` — NOT `sendBeacon` (can't set the header).
- **Jobs**: export/import ride the existing `startJob` single-slot (`src/auto-refresh.js`); campaigns get a mirrored own slot (network-bound; no gap-lock exposure); progress via the established poll idiom.
- **Web**: admin stays one lazy chunk (guest bundle unchanged); hash routing `#admin/<section>`; reuse Sheet/NumberInput/MultiSelect/ReorderList/DataTable/CollapseSection primitives + token theme; Recharts stays admin-only.

## Non-goals

- No prediction-logic changes; no ranking changes from analytics data (mine freely, ship skeptically — M4.x discipline unchanged).
- No third-party analytics (GA remains an optional VITE knob, not part of this program).
- No live deploy (user-gated, separate).
