## M14 — Scheduled maintenance mode
Settings keys (group `maintenance`, all `live:true`, EnvSchema fallbacks kept per decision 4): `MAINTENANCE_SCHEDULED` (bool, the admin toggle), `MAINTENANCE_START`/`MAINTENANCE_END` (EAT `YYYY-MM-DD HH:mm`, pattern-validated), `MAINTENANCE_MESSAGE` (default `We will have scheduled maintenance downtime from ${downtime_start} to ${downtime_end}`; closed placeholder set, unknown placeholders rejected at save).
Pure `src/db/maintenance-rules.js` (zero-import, shared VERBATIM server/web like magic-rules): `parseMaintenanceWindow(start, end)` (explicit `+03:00` offset — the KICKOFF_SQL_EXPR lesson, never bare-datetime local decode; invalid/reversed → null), `maintenanceState(cfg, nowMs)` → `'off'|'scheduled'|'active'` (past-end auto-expires to off), `renderMaintenanceNotice(template, window)` (total, default template on blank), `windowSignature(start,end)` (banner-dismiss key), `retryAfterSeconds(endMs, nowMs)`. Offline tests for all.
Server: one middleware before the routes — state `active` answers 503 UNLESS the request carries a valid admin session or `ADMIN_TOKEN`/`API_TOKEN` bearer, or targets `/api/auth/*` (admin sign-in mid-window). `/api/*` → 503 JSON `{error:'maintenance', maintenance:{state,start,end,message}}` + `Retry-After: <s-to-end>`; page requests → self-contained static 503 HTML (rendered message + window, meta-refresh ~60s — dev/Express topology only; prod Apache serves the SPA regardless, which is why the client switch below is the real prod surface). `GET /api/refresh` payload gains the same `maintenance` object (rides the existing 60s freshness poll — no new endpoint).
Client: cache the schedule in localStorage `oddspro.maintenance` on every poll; `state==='scheduled'` renders a thin dismissible warning banner ABOVE the nav bar (warning tint; dismissal stores `windowSignature` so the same window stays dismissed but an edited window re-surfaces). Own-clock timer auto-switches to a full-screen maintenance overlay at `start` WITHOUT reload: freshness polling stops, records fetches suspend, tracking flushes pause. At `end` (+ 5–30s random jitter against stampede/skew) auto-recover: silent reload + polls resume; if the API still answers a maintenance 503 (window extended) the fresh body re-enters the mode with the new window. `api.js` additionally intercepts ANY maintenance-503 response into the same switch (covers a client clock behind the server's).
Admin UX: "Maintenance" card on the Dashboard section (M5 shell): state chip (off/scheduled/active), the toggle, two datetime inputs, template textarea with live preview, and a "+1h from now" convenience preset — writes through the standard settings PUT (M6 audit trail dates every change).
Verify: short live window E2E — schedule 2 min out → banner on a guest client → dismiss persists across reload → auto-switch at start with NO refresh (network tab quiet) → API 503 + Retry-After for guests while an admin session passes → auto-recover after end; suite green.
