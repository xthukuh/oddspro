# M4.2b — live-ledger validation of the precursor boosters + the value-edge hunt (2026-07-16)

*Read-only. Three probes, all reproducible from named scripts (bottom). No
warehouse table written, no API called. Follows `docs/research/precursor-patterns.md`
(the warehouse mine that produced the three boosters) and
`docs/research/emergence-patterns-findings.md` (the tip-ledger mine). Built to the same

## One-paragraph truth

**Ship nothing.** The three "best-evidenced-but-never-shipped" precursor
boosters were re-tested on the LIVE data with the M4.2 recency/regime
discipline, and none survives at the level that matters. **B3 (U 3.5 support ≥
0.7)** raises U 3.5's raw market hit-rate a real, well-powered **+8.5pp** (and
+10.9pp on recent tips) — but when driven through the ACTUAL Safe pool it makes
the pool **−2.3pp worse**, because the pool selects almost no U 3.5 legs and the
few it does select already pass gates that make B3 redundant. **B1** is a real
signal priced sub-1.20 (unbettable); **B2** shows no discrimination among our
O 2.5 tips and is fading. Separately, a pre-registered **value-edge hunt** —
does `stats_support − devigged_market_prob` at a bettable price earn +EV? —
found **zero** CI-positive, BH-surviving slices across 14 markets × 5
thresholds, and the calibration test underneath it explains why: **our rolling
stats carry essentially no information orthogonal to the bookmaker's price**
(realized-minus-devig is flat noise across every value bin). The synthesis is a
**dual local optimum**: profit is impossible on the current signal set (stats =
market), and win-probability is already near-maxed (the best booster adds
nothing inside the Safe pool). The only remaining lever is **orthogonal signal**
— a feature that beats the market where rolling stats don't — which is exactly
what the dormant M4.1 AI-enrichment was built to supply.

## Recency/regime discipline (the M4.2 lesson, applied)

`TIP_MIN_PRICE` moved **1.20 → 1.35 on 2026-07-10**, splitting the settled-tip
ledger into two populations (per-day min tip price steps 1.20 → 1.29 → 1.35).
Every probe here detects that break automatically and **headlines the recent
regime** (2026-07-11 … 2026-07-16). The linked-with-odds surface (1,645 finished
fixtures, Jul 02–16) is NOT gated by `TIP_MIN_PRICE`, so it carries the power;
the tip ledger carries the decision-relevance.

## Probe 2 — does B3 survive to the REAL Safe pool? (the decisive test)

Simulated `src/db/magic-rules.js` `safeSelection` verbatim, per-day
leave-one-day-out calibration (the `simulateStrategies` idiom), baseline vs the
B3 hard eligibility gate.

| Recent regime (6 days) | legs | picks/day | leg-survival | slip-survival | U 3.5 legs |
|---|---|---|---|---|---|
| baseline | 18 | 3.00 | **61.1%** (11/18) | 16.7% (1/6) | 2 (both hit) |
| B3-gated | 17 | 2.83 | **58.8%** (10/17) | 16.7% (1/6) | 1 (hit) |
| **delta** | | | **−2.3pp** | +0.0pp | gate dropped 1 U 3.5 leg — *it had hit* |

Full window (14 days): baseline 69.2% → B3-gated 68.4% (**−0.8pp**); the gate
changed the selection on **1 of 14 days**.

**Why the market-level +10.9pp evaporates — structural, not sample noise:** the
Safe pool selects only **2 of 18** legs as U 3.5. It is dominated by short-price
double-chance legs the `sure` ranker + `minAgreement 0.65` + `maxPrice 1.6`
gates already prefer, and the U 3.5 legs that *do* survive those gates already
hit ~100%. B3's support filter is **collinear with the gates the pool already
applies** — it can only touch ~10% of legs, and when it fired it removed a
*winning* leg. The booster's lift lives on the raw, unfiltered market; inside
the Safe pool there is no residual signal for it to add. *(Caveat: the recent
pool is thin — 18 legs / 6 days — so this cannot detect a tiny benefit; but the
composition fact (U 3.5 ≈ 10% of the pool, already high-quality) makes a
material pool improvement impossible regardless of outcome sample.)*

## Probe 3 — the value-edge hunt (pre-registered, EXPLORATORY)

**The idea:** every sort this project ships selects on *high stats support*,
which `precursor-patterns.md` proved lands in the cheap sub-1.20 tail. **Value ≠
confidence.** Define `value = stats_support − devigged_market_prob`; a value tip
(stats 0.60, market 0.50) is priced ~1.9 (bettable) and pays only if our stats
know something the price does not. `precursor-patterns.md:167` explicitly left
this door open: *"a future 'value' sort would target mid-priced X2/O 2.5, not
high stats support."*

**H-CALIB — does `value` predict the outcome beyond the market?** Mean
(realized − devig) by value bin, 21,768 (fixture,market) rows over 15 days:

| our stats vs market | n | realized − devig |
|---|---|---|
| much *less* likely (value < −0.05) | 3,076 | +0.94pp |
| ≈ market (0 to +0.05) | 3,145 | −0.85pp |
| much *more* likely (value ≥ +0.10) | 5,220 | **+0.13pp** |

**Flat noise, uncorrelated with value.** If our stats added orthogonal
information the last row would climb hard positive; it sits at zero. This is the
mechanistic root of "no edge": you cannot beat a price with a number that equals
it.

**H-VALUE — bettable (≥1.20) positive-value slices, day-clustered EV CI,
Benjamini-Hochberg over all 14×5 swept slices:** best point estimates are
`1 value≥0.08` +4.5% and `NG value≥0.12` +2.8%, both with CIs spanning zero and
bootstrap p ≈ 0.3–0.4. **Zero slices are CI-positive AND BH-surviving.** The
mid-price value region is −EV too. The edge door is shut, with a reason nailed
to it.

## Verdict & forward path

- **Ship nothing to `magic-rules.js`.** All three boosters and the value sort
  fail the live gate. The Safe/`sure` pool is at the ceiling of what
  win-probability selection can extract from the current signal set, and no
  bettable +EV edge exists because our stats do not beat the market's price.
- **The only remaining lever is orthogonal signal.** `probe-value-edge.js` is
  now a **reusable edge-detector**: the moment a new feature exists (AI-derived
  injury/motivation/lineup facts — M4.1 enrichment, currently dry: 7 insight
  rows vs ~1,800 needed), run the identical value-calibration test to see
  whether *it* carries information the market has not priced. If any feature's
  high-value bin climbs where rolling stats' stayed flat, that is the first real
  candidate for an edge this project has ever had — subject, as always, to a
  pre-registered OOS confirmation before it ships.
