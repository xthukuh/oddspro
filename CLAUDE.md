## Project Overview

oddspro is a MySQL data warehouse for football bookmaker odds and stats. It scrapes odds from two Kenyan bookmakers (BetPawa, Betika), ingests canonical fixture/result/stats data from API-Football (api-sports.io), and correlates bookmaker matches to canonical fixtures via fuzzy matching with learned aliases. Plain Node.js (ES modules); knex/mysql2 for the DB layer; zod validates all external data.

Source spec and progress tracking live in `implementation-plan.md` (phases 1–28: warehouse, ingestion, linking, deep stats, visualization, stale odds, pre-match snapshots, hot picks/tips, magic sort, betslip playground, in-process auto-refresh, iOS UI). **Current release: v1.2.0 — tagged `v1.2.0` (`f7f1f9d`, 2026-07-17, suite 707/707) on `main`; deploy package built in `release/` (gitignored — zips + `DEPLOY-CHECKLIST-v1.2.0.md`). The LIVE host oddspro.ke still runs the pre-v1.1.0 build deployed 2026-07-12 (verified 2026-07-17: web titled v1.0.3, `/api/settings` 404) — the manual cPanel upload is the remaining step** (deploy is not automatic — see `docs/DEPLOYMENT.md`). The v1.0.1 UI-tweak + deploy-prep pass (phases A–J + Phase G) is tracked in `docs/v1.0.1-ui-tweaks-checklist.md`. **v1.0.2 (merged to `main` 2026-07-13, `1b4ff7e`, suite 360/360):** a web UI pass (`docs/v1.0.2-ui-pass-checklist.md`) plus visitor analytics + geo backfill + `/admin` dashboard, a Select-header bulk-actions dropdown, portable config export/import, and the apisports null-event fix + results 7-day age cap — detailed in the Architecture notes below. **v1.0.3 (merged to `main` 2026-07-13, suite 365/365, root+web `1.0.3`):** a data-driven prediction-accuracy pass — the `sure` "Most likely to win" default sort (live-shrunk `safePrior` × confidence) + an auto-applied stats-sufficiency "hide risky games" gate + three risk tiers, grounded in a warehouse backtest + live-odds cross-validation + a 5-agent adversarial review. **Honest headline: no positive-EV market exists on our books (≈ −3% flat-stake EV); `sure` maximizes win probability, not profit; no market expansion shipped (team totals/BTTS are −EV on real odds).** Full study in `docs/sure-win-analysis.md`. **v1.1.0 (merged to `main` 2026-07-15, `d926b6b`, suite 478/478 — live deploy pending):** user accounts + tiered access, tracked in `docs/v1.1.0-implementation-plan.md`. Phone + 4-digit PIN auth (peppered scrypt hashes, opaque hashed DB sessions, SMS OTP verification via Bonga — dev mode logs the code), an in-app admin settings editor over a curated dynamic-settings catalog, an admin data-viz lab (Recharts, pre-binned aggregates only), cross-device prefs sync (LWW), **guest gating on `/api/records`** (no future dates, reasoning redacted server-side — this RESOLVES the v1.0.1 note "gate server-side for real secrecy"), and response caching (data_version-keyed memo + gzip + ETag/304) on the heavy read endpoints.

`[date]` defaults to today; accepts anything `new Date()` parses, or `today`/`now`. All actions are idempotent and cron-able — a Windows Task Scheduler task `oddspro-pipeline` runs `scripts/pipeline-task.cmd` (full sweep → `logs/pipeline.log`) daily at 08:00. There is no linter. Restart `npm run serve` after pulling backend changes — a stale server process holds :3001 with old code.

## Architecture

Pipeline: **odds scrapers + fixtures ingester → MySQL warehouse → linker correlates → results settle → deep stats accumulate**.

- `src/index.js` — CLI dispatcher; every action closes the shared knex pool on exit. No action (or `start`/a bare day count) runs `src/pipeline.js`.
- `src/pipeline.js` — default full sweep (`npm run start`), ordered for fewest server hits: fixtures per date first (a date fetch also refreshes today's statuses, shrinking the results refresh set), then results (completes matches so scrapers skip their per-game detail requests via the `completedMatchIds()` exclusion set), then odds per provider per date, then a single link pass, then stats + standings + team history + pre-match snapshots. `runStartPipeline(days, onStep)` narrates steps and returns `{ dates, quota_remaining }` (the scheduler's full mode reuses it verbatim). Also exports `runDateRefresh(date, onStep)` — the single-date subset behind the web refresh button (skips results/stats for future dates, history/snapshots for past dates; standings stays owned by the full sweep). `runDateRefresh` also deliberately EXCLUDES the M4.1 AI-enrichment step (`enrichFixtures()`, full-sweep-only, step 12 below) — a cost boundary, not an oversight: the web refresh button can be triggered far more often than the daily full pipeline, and enrichment bills real per-fixture AI calls.
- `src/auto-refresh.js` — in-process scheduler + the single-slot refresh job state shared with the manual endpoint (one guard, auto and manual can never overlap). One unref'd 30s `setInterval` tick: full sweep due once per EAT day at `AUTO_FULL_AT` (stamped at START — one attempt/day even on failure; a mid-day restart never fires a surprise sweep), else light pass every `AUTO_LIGHT_MINUTES` — `lightRefresh()`: `settleApisportsResults()` (statuses/scores/`elapsed` per id, no whole-date fetch) → today's odds per provider (completed excluded) → `linkMatches()` → `settleHotPicks()`. Successful completions bump a monotonic `data_version` + `last_success {at,mode,dates}` (the client freshness signal) and stamp a success-only per-date freshness map (`lastFreshAt`) — deliberately separate from server.js's manual cooldown map (a 10-min cadence stamping the cooldown would keep today permanently 429'd). Per-job lines land in self-truncating `logs/auto-refresh.log` (`AUTO_LOG_MAX_KB`). Pure schedule math (EAT due-times, `trimLogTail`) lives in zero-import `src/db/auto-rules.js` (offline-tested).
- `src/betpawa.js` / `src/betika.js` — bookmaker scrapers (browser-mimicking axios clients against undocumented public APIs). Both emit the same standardized game record consumed by `store.saveMatches()`.
- `src/apisports.js` — API-Football client + all its fetchers. Quota-guarded (`x-ratelimit-requests-remaining` header, halts at `APISPORTS_MIN_REMAINING` floor), paginated, zod-validated. The per-minute burst budget (`x-ratelimit-remaining` header) is paced, not fatal: `_getPage` sleeps into the next minute window when it runs out and retries an `errors.rateLimit` response (bounded, 2 max) — decision logic in pure `src/db/rate-rules.js` (offline-tested; the daily-quota error deliberately does NOT match and stays fatal). Fixtures are fetched with `timezone=Africa/Nairobi` so kickoffs align with bookmaker wall-clock times. Event parsing is delegated to the tolerant `src/apisports-events.js` (survives `type:null` from live `/fixtures/events`; a per-fixture ZodError is caught and skipped, quota/network stays fatal). The results refresh only re-polls non-terminal past-kickoff fixtures within `RESULTS_MAX_AGE_DAYS` (7) — stuck upstream `NS`/`PST` zombies older than that are retired from the poll set (a reschedule moves kickoff forward and re-enters naturally).
- `src/link.js` — correlation. API-Football fixtures are the **canonical base record**; `matches.fixture_id` is the link. Provider order matters: betpawa first, betika last (betika lacks identifier attributes and additionally scores against betpawa matches already linked to a candidate fixture). Fuzzy scorer = best of bigram-Dice / token-set / overlap-coefficient / initialism over normalized names; competition similarity is a bonus (+0.1×sim), never a veto; acceptance needs `LINK_MIN_CONFIDENCE` (default 0.85) plus a 0.05 margin over the runner-up. Confident links cache `team_aliases` / `league_aliases` for instant future correlation.
- `src/db/connection.js` — the only knex instance (never use raw mysql2). Session `time_zone` is pinned to +03:00 in `knexfile.js` so SQL `NOW()` compares correctly against stored EAT wall-clock datetimes.
- `src/db/store.js` — odds persistence: upsert `matches` by `(provider, provider_match_id)` (with an explicit `updated_at` bump — MySQL's ON UPDATE skips no-op updates), then refresh `odds_markets` via `src/db/odds-diff.js`: markets present in the latest snapshot are replaced (delete+insert), vanished ones are kept flagged `is_stale` (last-seen price survives for display), re-listed markets revive. Never touches `completed_at`/`fixture_id` (owned by results/link); `matches.metadata` (raw ~39 KB/row provider JSON, no readers) is **insert-only** since the 2026-07-17 perf pass — first-sight blob kept forever, refreshes no longer rewrite it (it alone is 556 MB; reclaiming the historical blobs stays user-gated).
- `src/prematch.js` — pre-match snapshot writer (`updatePrematchSnapshots()`): upserts `fixture_prematch` (rank/form/H2H/rolling-goals aggregates) for every upcoming correlated fixture; the `kickoff > NOW()` selection **is** the freeze — past fixtures are never selected again, so their last pre-kickoff snapshot stands forever. Single-statement chunked upserts (no delete+insert, no deadlock exposure).
- `src/hotpicks.js` — over-2.5 hot picks + best tips (`updateHotPicks()`): settles `fixture_predictions` outcomes via the standalone `settleHotPicks()` export (hot picks via SQL, tips via pure `tipOutcome` — three buckets hit/miss/**`void`** (a DNB push on a draw); cheap, no fetches/AI, also called by the light auto-refresh pass) from canonical final scores, then re-evaluates every upcoming correlated snapshot-backed fixture through the deduction gates (**M3: `_loadMarkets` slims to SQL + per-fixture row grouping and delegates to tip-rules' `buildTipBooks`, so tips now span all seven families**). `loadTeamHistory(teamIds)` (exported, M4.1 final review finding 1) is the shared bulk finished-fixtures loader grouped per team — src/enrich.js's AI-enrichment rolling-stats projection reuses it VERBATIM rather than a second, drifting definition of "recent form". **Detour A (2026-07-17): the sweep itself bills NO AI — the 8 verdict columns (`ai_*`/`tip_ai_*`) are WORKER-OWNED** (absent from the sweep's upsert merge, the settle-columns idiom; the sweep keeps only a read-only reusable-veto re-apply on `hot`), drained by `src/ai-worker.js` (60s serve tick + `node src/index.js aireview`). Rule-hot candidates optionally pass a Gemini confirm/veto (`src/ai/adjudicators.js` via the `callStructured` harness, fail-open — errors keep the rule verdict; verdicts are reused while the judged score/tip identity AND the `aiModelTag()` are unchanged per pure `src/db/adjudicate-rules.js` — verdict-time context travels as `judged` inside the review JSON — so reruns don't re-bill and model/grounding/prompt-version switches re-adjudicate automatically; the tag carries a `#p<N>` prompt-version suffix). Adjudicator prompts are v2 (2026-07-05, de-anchored): the model researches context + team news (google_search when `HOTPICK_AI_WEB=1`), states its OWN probability, and vetoes mechanically only below break-even − 0.05 or on a verified disqualifier — market-vs-stats contradiction alone is never a veto ground (v1's contradiction vetoes were net-negative). Structured verdicts (probability, per-check findings, grounding citations) persist to `ai_review`/`tip_ai_review` JSON; reply decoding lives in pure `src/ai-parse.js` (zod-only, offline-testable). Tips run `tipEligibility` first (thin-evidence fixtures store `tip_skip_reason` instead of a market-only tip), persist the full `bestTip` blend as `tip_breakdown` JSON, and the top-confidence tips (`TIP_AI_MIN_CONFIDENCE`, capped `TIP_AI_DAILY_CAP`/run) get an AI review (`reviewTip`, same reuse idiom keyed on tip market+price) — a tip veto FLAGS the row (`tip_ai_verdict`) but never clears it, so its settled outcome measures what the veto was worth; `hot=false` on a hot-pick veto (`if (verdict.verdict === 'veto') c.row.hot = false`) is UNCHANGED — the AI can still veto a hot pick, never promote one. **The tip veto no longer shapes tip RANKING or DISPLAY** (M4.1 spec §3.8: settled data shows no discrimination, confirm 75.0% vs veto 72.7% at n=61) — `magicSortRows`/`scoreTip` (`src/db/magic-rules.js`) deliberately do NOT consult `tip_ai_verdict`, the web table no longer strikes a vetoed tip through, and the betslip pool (`BetslipPlayground.jsx`) no longer filters on it either; the verdict is still PERSISTED and surfaced everywhere it was before — the details-gated `TipPopover.jsx` "AI double-check" section, the web's selection "Export CSV" (`Tip AI Verdict` column), and `perf-rules.js`'s `ai_impact` bucket (vetoed picks settle but are excluded from the headline rate, so `saved` still measures what following the vetoes was worth). Same freeze idiom as prematch: `kickoff > NOW()` selection, chunked onConflict-merge upsert (merge list excludes `result_goals`/`outcome`/`tip_outcome` — owned by the settle pass). `hotpicksSummary()` backs `GET /api/hotpicks`; `performanceSummary()` (thin loader over pure `src/db/perf-rules.js`) backs `GET /api/performance` + the `performance` CLI report.
- `src/enrich.js` + `src/ai/` + `src/db/ai-rules.js` — M4.1 AI enrichment (`enrichFixtures()`, wired as `node src/index.js enrich` and pipeline step 12, deliberately AFTER hot picks so the anchored call sees the tip `updateHotPicks()` just wrote). **COLLECTION ONLY — nothing here feeds `bestTip`, confidence or any ranking**; that is a later milestone, answered by replay, never by assertion. OFF by default (`AI_ENRICH_ENABLED`); when on, each upcoming correlated fixture (bounded by `AI_ENRICH_CAP` fixtures/run, `AI_ENRICH_CONCURRENCY` in-flight) gets up to three calls: **facts** (Gemini, grounded, once per fixture — typed availability/motivation/congestion/lineup facts, absent evidence stays `null` rather than guessed), **blind** (a pinned NON-GOOGLE reasoner via `src/ai/openrouter.js` — `resolveTask` in `src/db/ai-rules.js` throws at boot if `AI_BLIND_MODEL`/`OPENROUTER_MODEL` resolves to a Google model, since reasoner independence is what the consensus signal rests on — sees the same screened facts, NO odds/tip), and **anchored** (Gemini again, sees everything including our tip + its price; only fires once a tip exists). Blind and anchored work IDENTICAL evidence, so `anchored − blind` is a paired anchoring-effect measurement — the stats block both prompts see (spec §3.1: "teams, date, our rolling stats") is `src/hotpicks.js`'s shared `loadTeamHistory` + `pairedTeamGoalsAggregates`/`h2hGoalsAggregates` (`src/db/goals-rules.js`, same fairness-paired windows the hot-pick evaluator itself uses — one warehouse machinery, not a second definition of "recent form"), scoped to only the run's capped targets; a genuinely sample-less side/H2H is OMITTED from the prompt rather than rendering literal `null`s (`ai-rules.js`'s `_team`, pure). `src/ai/index.js` is the provider seam (`callModel({task,prompt,cfg})`, mirrors the SMS seam) over `src/ai/gemini.js`/`src/ai/openrouter.js` — both now expose `enabled()` (uniform seam interface); `enrichFixtures()` fails fast, before any fixture is touched, on a misconfigured blind model (`resolveTask`), an unset `GEMINI_API_KEY` or an unset `OPENROUTER_API_KEY` (`assertAiProvidersConfigured` — closes the "anchored-with-no-blind on every fixture" failure mode a missing OpenRouter key used to cause). `resolveTask`/`callModel` resolve their task->model routing through `effectiveAiConfig()` (config defaults layered with `settings.effectiveConfig()`'s live admin overrides), so `OPENROUTER_MODEL`/`AI_BLIND_MODEL`/`AI_ANCHORED_MODEL` — SETTINGS_CATALOG `live:true` entries — actually take effect without a restart, same as every other admin-editable knob. `PROMPT_VERSION` is 2 (bumped when real stats replaced the hardcoded placeholder — reuse is keyed on the tag, so the fix correctly re-enriches upcoming fixtures). Same freeze idiom as prematch/tips/hot picks — `kickoff > NOW()`, enforced BOTH in SQL and in the pure, test-asserted `selectEnrichable()` guard, via the `+03:00`-offset-qualified `KICKOFF_SQL_EXPR` (a bare `f.kickoff` column decodes in the NODE PROCESS's local timezone, not the pinned SQL session — an off-EAT host could otherwise admit an already-kicked-off fixture). **This is exactly why AI reviews can never be backfilled: a grounded call on a played fixture retrieves the final score off the web, so collection is strictly wall-clock-bound and forward-only, never retroactive.** Reuse is keyed on `(fixture, kind, provider, model_tag)` plus the stored tip identity for `anchored` rows (`insightIsFresh`) — a rerun re-bills nothing, but a changed model/grounding/prompt-version OR a tip `bestTip` re-picked re-fires automatically. ### Key invariants

- **Fetch throttling:** `matches.completed_at` set ⇒ odds refreshes skip the match. Fixtures reaching a terminal status (`FT/AET/PEN/AWD/WO/CANC/ABD`) complete their linked matches; unlinked matches complete 4h after start (fallback).
- **Fetch-once stats:** `fixtures.stats_fetched_at`/`lineups_fetched_at`/`events_fetched_at` guarantee each final fixture costs at most 3 detail requests ever. Empty responses only set the flag after 48h post-kickoff (minor leagues may never publish stats). Never delete or refetch immutable API data.
- **Pre-match snapshots freeze at kickoff:** `fixture_prematch` rows are upserted every run while the fixture is upcoming and never written after kickoff — historical pre-match stats must stay exactly as they were, unaffected by later matches.
- **Hot picks freeze at kickoff and settle exactly once:** `fixture_predictions` rows use the same `kickoff > NOW()` selection-freeze; the settle pass (canonical FT scores → `result_goals`/`outcome`) is the only writer of those columns. The hit-rate scoreboard is honest by construction — never rewrite a settled or past-kickoff pick. AI adjudication is optional (no `GEMINI_API_KEY` = rules-only) and can only veto, never promote; API `/predictions` uses the fetch-once flag `fixtures.predictions_fetched_at`.
- **Fetch-once history:** `fixtures.history_fetched_at` guarantees each upcoming correlated fixture costs at most 3 backfill requests ever (2× team last-N, 1× full head-to-head). Only FINAL_STATUSES items are saved — headtohead also returns future meetings, which must not leak into the results refresh set.
- **Results are canonical:** the `results` action copies authoritative scores from final fixtures into linked matches (bookmaker-parsed scores are unreliable for upcoming games — BetPawa reports 0-0, Betika null).
- **Betika null fields:** `home_team_id, away_team_id, region_id, region_name, category_id, competition_id` are always null (not exposed by its API).
- **Access tiers are server-authoritative (v1.1.0):** guest redaction/date-gating happens in `records.js`/`server.js`, never client-only; the response memo key MUST carry the tier (a guest and a full body can never share a cache slot). `AUTH_ENABLED=0` and machine bearers are the legacy full-access path (`access=null`). Sessions store only sha256 hashes — a DB leak yields no usable token; rotating `PIN_PEPPER` invalidates every stored PIN (deliberate global reset, never do it casually).
- Migrations are forward-only; never edit an applied migration.

## Conventions

- ES modules, `async/await`, 4-space indentation (workspace setting — overrides the usual 2-space Node default), single quotes, semicolons.
- Foreign keys use `ON DELETE CASCADE`/`RESTRICT`; a NULLABLE audit pointer may use `ON DELETE SET NULL` (prior art: `settings.updated_by`, batch 11 — deleting a user must not delete the settings they once touched). Migrations that seed data may read `process.env` directly (they run under the knex CLI as well as the app; prior art: the users migration's `ADMIN_SEED_PIN`/`PIN_PEPPER`).
- All external data (API responses, env) through zod schemas; keep field schemas tolerant (`nullable().optional()`) — live data has taught this (`league.round` can be null; `/fixtures/events` `type` can be null — parsed by the tolerant `src/apisports-events.js`, and per-fixture ZodErrors are caught so one bad record can't abort a sweep).
- `x-*-output.xx.json` files at the root are legacy fetched-data snapshots — do not delete.
