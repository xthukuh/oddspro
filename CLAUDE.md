## Project Overview

oddspro is a MySQL data warehouse for football bookmaker odds and stats. It scrapes odds from two Kenyan bookmakers (BetPawa, Betika), ingests canonical fixture/result/stats data from API-Football (api-sports.io), and correlates bookmaker matches to canonical fixtures via fuzzy matching with learned aliases. Plain Node.js (ES modules); knex/mysql2 for the DB layer; zod validates all external data.

Source spec and progress tracking live in `implementation-plan.md` (phases 1‚Äì28: warehouse, ingestion, linking, deep stats, visualization, stale odds, pre-match snapshots, hot picks/tips, magic sort, betslip playground, in-process auto-refresh, iOS UI). **Current release: v1.0.1 ‚Äî merged to `main` (`cde3772`) and LIVE at oddspro.ke since 2026-07-12** (manual cPanel upload; deploy is not automatic ‚Äî see `docs/DEPLOYMENT.md`). The v1.0.1 UI-tweak + deploy-prep pass (phases A‚ÄìJ + Phase G) is tracked in `docs/v1.0.1-ui-tweaks-checklist.md`. **v1.0.2 (merged to `main` 2026-07-13, `1b4ff7e`, suite 360/360):** a web UI pass (`docs/v1.0.2-ui-pass-checklist.md`) plus visitor analytics + geo backfill + `/admin` dashboard, a Select-header bulk-actions dropdown, portable config export/import, and the apisports null-event fix + results 7-day age cap ‚Äî detailed in the Architecture notes below. **v1.0.3 (merged to `main` 2026-07-13, suite 365/365, root+web `1.0.3`):** a data-driven prediction-accuracy pass ‚Äî the `sure` "Most likely to win" default sort (live-shrunk `safePrior` √ó confidence) + an auto-applied stats-sufficiency "hide risky games" gate + three risk tiers, grounded in a warehouse backtest + live-odds cross-validation + a 5-agent adversarial review. **Honest headline: no positive-EV market exists on our books (‚âà ‚àí3% flat-stake EV); `sure` maximizes win probability, not profit; no market expansion shipped (team totals/BTTS are ‚àíEV on real odds).** Full study in `docs/sure-win-analysis.md`. **v1.1.0 (merged to `main` 2026-07-15, `d926b6b`, suite 478/478 ‚Äî live deploy pending):** user accounts + tiered access, tracked in `docs/v1.1.0-implementation-plan.md`. Phone + 4-digit PIN auth (peppered scrypt hashes, opaque hashed DB sessions, SMS OTP verification via Bonga ‚Äî dev mode logs the code), an in-app admin settings editor over a curated dynamic-settings catalog, an admin data-viz lab (Recharts, pre-binned aggregates only), cross-device prefs sync (LWW), **guest gating on `/api/records`** (no future dates, reasoning redacted server-side ‚Äî this RESOLVES the v1.0.1 note "gate server-side for real secrecy"), and response caching (data_version-keyed memo + gzip + ETag/304) on the heavy read endpoints.

`[date]` defaults to today; accepts anything `new Date()` parses, or `today`/`now`. All actions are idempotent and cron-able ‚Äî a Windows Task Scheduler task `oddspro-pipeline` runs `scripts/pipeline-task.cmd` (full sweep ‚Üí `logs/pipeline.log`) daily at 08:00. There is no linter. Restart `npm run serve` after pulling backend changes ‚Äî a stale server process holds :3001 with old code.

`.env` (gitignored, see `.env.example`) holds MySQL credentials (`DB_HOST/DB_PORT/DB_DATABASE/DB_USERNAME/DB_PASSWORD/DB_CHARSET/DB_COLLATION` ‚Äî Laravel-style names), API-Football credentials (`X_APISPORTS_URL`, `X_APISPORTS_KEY`), and optionally `GEMINI_API_KEY` (+ `HOTPICK_AI_MODEL` default `gemini-2.5-flash`, `HOTPICK_AI_WEB` ‚Äî attaches Gemini's `google_search` grounding tool, `TIP_AI_MIN_CONFIDENCE`/`TIP_AI_DAILY_CAP`, `HOTPICK_*`/`TIP_MIN_*` threshold overrides) for the AI adjudicators (Google Gemini; replaced OpenRouter 2026-07-04) ‚Äî the key lives ONLY in `.env`, never in git. ## Architecture

Pipeline: **odds scrapers + fixtures ingester ‚Üí MySQL warehouse ‚Üí linker correlates ‚Üí results settle ‚Üí deep stats accumulate**.

- `src/index.js` ‚Äî CLI dispatcher; every action closes the shared knex pool on exit. No action (or `start`/a bare day count) runs `src/pipeline.js`.
- `src/pipeline.js` ‚Äî default full sweep (`npm run start`), ordered for fewest server hits: fixtures per date first (a date fetch also refreshes today's statuses, shrinking the results refresh set), then results (completes matches so scrapers skip their per-game detail requests via the `completedMatchIds()` exclusion set), then odds per provider per date, then a single link pass, then stats + standings + team history + pre-match snapshots. `runStartPipeline(days, onStep)` narrates steps and returns `{ dates, quota_remaining }` (the scheduler's full mode reuses it verbatim). Also exports `runDateRefresh(date, onStep)` ‚Äî the single-date subset behind the web refresh button (skips results/stats for future dates, history/snapshots for past dates; standings stays owned by the full sweep).
- `src/auto-refresh.js` ‚Äî in-process scheduler + the single-slot refresh job state shared with the manual endpoint (one guard, auto and manual can never overlap). One unref'd 30s `setInterval` tick: full sweep due once per EAT day at `AUTO_FULL_AT` (stamped at START ‚Äî one attempt/day even on failure; a mid-day restart never fires a surprise sweep), else light pass every `AUTO_LIGHT_MINUTES` ‚Äî `lightRefresh()`: `settleApisportsResults()` (statuses/scores/`elapsed` per id, no whole-date fetch) ‚Üí today's odds per provider (completed excluded) ‚Üí `linkMatches()` ‚Üí `settleHotPicks()`. Successful completions bump a monotonic `data_version` + `last_success {at,mode,dates}` (the client freshness signal) and stamp a success-only per-date freshness map (`lastFreshAt`) ‚Äî deliberately separate from server.js's manual cooldown map (a 10-min cadence stamping the cooldown would keep today permanently 429'd). Per-job lines land in self-truncating `logs/auto-refresh.log` (`AUTO_LOG_MAX_KB`). Pure schedule math (EAT due-times, `trimLogTail`) lives in zero-import `src/db/auto-rules.js` (offline-tested).
- `src/betpawa.js` / `src/betika.js` ‚Äî bookmaker scrapers (browser-mimicking axios clients against undocumented public APIs). Both emit the same standardized game record consumed by `store.saveMatches()`.
- `src/apisports.js` ‚Äî API-Football client + all its fetchers. Quota-guarded (`x-ratelimit-requests-remaining` header, halts at `APISPORTS_MIN_REMAINING` floor), paginated, zod-validated. The per-minute burst budget (`x-ratelimit-remaining` header) is paced, not fatal: `_getPage` sleeps into the next minute window when it runs out and retries an `errors.rateLimit` response (bounded, 2 max) ‚Äî decision logic in pure `src/db/rate-rules.js` (offline-tested; the daily-quota error deliberately does NOT match and stays fatal). Fixtures are fetched with `timezone=Africa/Nairobi` so kickoffs align with bookmaker wall-clock times. Event parsing is delegated to the tolerant `src/apisports-events.js` (survives `type:null` from live `/fixtures/events`; a per-fixture ZodError is caught and skipped, quota/network stays fatal). The results refresh only re-polls non-terminal past-kickoff fixtures within `RESULTS_MAX_AGE_DAYS` (7) ‚Äî stuck upstream `NS`/`PST` zombies older than that are retired from the poll set (a reschedule moves kickoff forward and re-enters naturally).
- `src/link.js` ‚Äî correlation. API-Football fixtures are the **canonical base record**; `matches.fixture_id` is the link. Provider order matters: betpawa first, betika last (betika lacks identifier attributes and additionally scores against betpawa matches already linked to a candidate fixture). Fuzzy scorer = best of bigram-Dice / token-set / overlap-coefficient / initialism over normalized names; competition similarity is a bonus (+0.1√ósim), never a veto; acceptance needs `LINK_MIN_CONFIDENCE` (default 0.85) plus a 0.05 margin over the runner-up. Confident links cache `team_aliases` / `league_aliases` for instant future correlation.
- `src/db/connection.js` ‚Äî the only knex instance (never use raw mysql2). Session `time_zone` is pinned to +03:00 in `knexfile.js` so SQL `NOW()` compares correctly against stored EAT wall-clock datetimes.
- `src/db/store.js` ‚Äî odds persistence: upsert `matches` by `(provider, provider_match_id)` (with an explicit `updated_at` bump ‚Äî MySQL's ON UPDATE skips no-op updates), then refresh `odds_markets` via `src/db/odds-diff.js`: markets present in the latest snapshot are replaced (delete+insert), vanished ones are kept flagged `is_stale` (last-seen price survives for display), re-listed markets revive. Never touches `completed_at`/`fixture_id` (owned by results/link).
- `src/prematch.js` ‚Äî pre-match snapshot writer (`updatePrematchSnapshots()`): upserts `fixture_prematch` (rank/form/H2H/rolling-goals aggregates) for every upcoming correlated fixture; the `kickoff > NOW()` selection **is** the freeze ‚Äî past fixtures are never selected again, so their last pre-kickoff snapshot stands forever. Single-statement chunked upserts (no delete+insert, no deadlock exposure).
- `src/hotpicks.js` ‚Äî over-2.5 hot picks + best tips (`updateHotPicks()`): settles `fixture_predictions` outcomes via the standalone `settleHotPicks()` export (hot picks via SQL, tips via pure `tipOutcome` ‚Äî three buckets hit/miss/**`void`** (a DNB push on a draw); cheap, no fetches/AI, also called by the light auto-refresh pass) from canonical final scores, then re-evaluates every upcoming correlated snapshot-backed fixture through the deduction gates (**M3: `_loadMarkets` slims to SQL + per-fixture row grouping and delegates to tip-rules' `buildTipBooks`, so tips now span all seven families**); rule-hot candidates optionally pass a Gemini confirm/veto (`src/ai.js`, fail-open ‚Äî errors keep the rule verdict; verdicts are reused while the score AND the `aiModelTag()` are unchanged, so reruns don't re-bill and model/grounding/prompt-version switches re-adjudicate automatically ‚Äî the tag carries a `#p<N>` prompt-version suffix). Adjudicator prompts are v2 (2026-07-05, de-anchored): the model researches context + team news (google_search when `HOTPICK_AI_WEB=1`), states its OWN probability, and vetoes mechanically only below break-even ‚àí 0.05 or on a verified disqualifier ‚Äî market-vs-stats contradiction alone is never a veto ground (v1's contradiction vetoes were net-negative). Structured verdicts (probability, per-check findings, grounding citations) persist to `ai_review`/`tip_ai_review` JSON; reply decoding lives in pure `src/ai-parse.js` (zod-only, offline-testable). Tips run `tipEligibility` first (thin-evidence fixtures store `tip_skip_reason` instead of a market-only tip), persist the full `bestTip` blend as `tip_breakdown` JSON, and the top-confidence tips (`TIP_AI_MIN_CONFIDENCE`, capped `TIP_AI_DAILY_CAP`/run) get an AI review (`reviewTip`, same reuse idiom keyed on tip market+price) ‚Äî a tip veto FLAGS the row (`tip_ai_verdict`) but never clears it, so its settled outcome measures what the veto was worth. Same freeze idiom as prematch: `kickoff > NOW()` selection, chunked onConflict-merge upsert (merge list excludes `result_goals`/`outcome`/`tip_outcome` ‚Äî owned by the settle pass). `hotpicksSummary()` backs `GET /api/hotpicks`; `performanceSummary()` (thin loader over pure `src/db/perf-rules.js`) backs `GET /api/performance` + the `performance` CLI report.
- `src/db/goals-rules.js` ‚Äî pure hot-pick deduction rules (zero imports, offline-testable): vig-removed `impliedProbability`, kickoff-cutoff team/H2H goal aggregates, API-Football prediction signal parsing, and `scoreOverLine(inputs, line)` ‚Äî the line-parameterized hot core (`scoreOver25` = the `line=2.5` case, byte-compat) with strict AND concurrence gates (both teams' rolling avg-total + over-rate, market probability floor, H2H veto, API veto), precision over recall by design. **M3:** `teamGoalsAggregates`/`h2hGoalsAggregates` carry per-line `overRates` and `apiPredictionSignal(pred, line)` is line-aware; `LINE_THRESHOLDS` gates which O/U lines may EVER fire hot ‚Äî the 2026-07-16 line sweep (1.5/2.5/3.5) found no line beats the 2.5 ~73% bar, so it ships `{2.5}` only (`HOTPICK_LINES=2.5`, an honest no-expansion). **Fairness pairing** (`pairedTeamGoalsAggregates`, mirrored by tip-rules' `pairedTeamOutcomeAggregates`): both teams of a fixture are judged over the SAME window, capped at the smaller side's qualifying count ‚Äî mixed windows bias the rate gates; `pool` keeps the uncapped counts for eligibility attribution. `DEFAULT_THRESHOLDS` tuned by `scripts/backtest-hotpicks.js` (54.3% baseline ‚Üí 73.3% stats-only precision over 10,678 fixtures, re-validated under pairing 2026-07-04); `HOTPICK_TEAM_WINDOW` must stay ‚â§ ~8 (history backfill only fetches 10 games/team).
- `src/db/tip-rules.js` ‚Äî pure "Tip" column deduction (zero imports): `teamOutcomeAggregates`/`h2hOutcomeAggregates` (W/D/L + per-line over rates, kickoff cutoff), `tipEligibility` ‚Äî evidence screen run BEFORE `bestTip` (context gate FIRST: `TIP_CONTEXT_EXCLUDE` skips friendly/youth/reserve leagues outright ‚Äî rolling form is invalid evidence there, the epicenter of the first settled losses; then both teams' sample ‚â• `minGames` + at least one full market group; a market-only blend is just the bookmaker's own devigged opinion and can't beat the vig), `bestTip` ‚Äî safest bettable outcome across 1X2/double-chance/every O/U line **plus (M3) BTTS (`GG`/`NG`), draw-no-bet (`DNB1`/`DNB2`), side-specific team totals (`TT:H/A:O/U <line>`) and odd/even (`ODD`/`EVEN`) ‚Äî the spec's `TIP_FAMILIES` registry realized as per-family candidate blocks (each with its own stats blend: BTTS = mean of both teams' + H2H `bttsRate`, DNB = win-vs-win renormalized `statsProb`, TT = scored-vs-conceded per-side over-rates, odd/even = `oddRate`) sharing `bestTip`'s `consider()` + one `tipOutcome` settle case; a new family is one block + one case**, confidence = renormalized blend of devigged market probability (0.6) + rolling-stats support (0.3) + API-Football percentages (0.1), with a price floor (`TIP_MIN_PRICE`, default 1.2) that excludes near-certain junk odds ‚Äî what survives at high confidence is the "hidden gem" ‚Äî and an Under line floor (`TIP_MIN_UNDER_LINE`, default 4.5: no U 2.5/U 3.5 tips ‚Äî near-Unders realized 61.9% vs 78.1% break-even; suppressed Unders yield to the runner-up); the return carries the justification breakdown (component probs, effective weights, samples, runners-up, winning `book_overround`) persisted verbatim as `tip_breakdown`. **Settlement (M3): `tipOutcome(market, fh, fa) ‚Üí 'hit'|'miss'|'void'`** (DNB draw = `void`/stake-returned; throws on unknown keys ‚Äî a persisted junk key is a loud bug), with `tipHit` the byte-compatible boolean wrapper (`=== 'hit'`) and `tipHitSafe` the never-throw browser variant (unknown ‚Üí `null`). **Book-integrity guards (M3, spec ¬ß4.2):** `bookIntegrity(prices)` screens each family book for completeness + an overround inside `TIP_MIN/MAX_OVERROUND` (below 1.0 smells like a palpable-error/boosted price; far above is margin loading that ruins the devig), and `selectFamilyBook` picks ONE provider's full family book (betpawa‚Üíbetika, never mixing providers inside a group) with a cross-provider `TIP_MAX_BOOK_DIVERGENCE` devig-divergence veto; `buildTipBooks(oddsRows, {homeName,awayName})` (imports M2's `canonicalMarket` ‚Äî sanctioned cross-pure import) assembles the per-family/per-line books from one fixture's FT-only rows (period-tagged excluded, TT side resolved by `tt.side` or normalized team-name match, unresolved dropped), returning `overrounds` + a `rejects` audit map. **These guards apply to the NEW families only** (BTTS/DNB/TT/odd-even, routed through `selectFamilyBook`) ‚Äî the canonical 1X2/double-chance/O-U candidate path deliberately bypasses both the overround band and the divergence veto (no `book_overround` on those candidates) to stay byte-compatible with the pre-M3 output. Tips live on `fixture_predictions` (`tip_market VARCHAR(32)`/`tip_price/tip_confidence/tip_outcome ENUM('hit','miss','void')` + `tip_breakdown/tip_skip_reason/tip_ai_*`; migration `20260715000001_tip_market_v2`), freeze at kickoff, sort/filter via the `tip` base column (confidence).
- `src/db/perf-rules.js` ‚Äî pure betting-performance calc (zero imports): `summarizePerformance(rows)` ‚Üí flat-stake ROI (1 unit/settled pick), hit-rate, avg price + break-even rate per 7d/30d/all window, buckets by confidence band / market group / exact O/U line (tips only ‚Äî keeps the near-Under and tail-Over residuals measurable) / edge sign (`confidence √ó price ‚àí 1`, the EV proxy), and AI-veto impact (vetoed picks settle but are excluded from headlines ‚Äî `saved` measures what following the vetoes was worth). Hit-rate alone can't prove profit; the edge buckets are where false positives are located.
- `src/db/magic-rules.js` ‚Äî pure magic-sort engine (imports only the perf-rules bucket labelers, still offline-testable) shared VERBATIM by the server loader and the web table (vite `server.fs.allow ['..']` serves the out-of-root import in dev; the build follows relative imports natively ‚Äî one scorer, no client/server drift): `computeCalibration` digests every settled tip into empirical hit-rate buckets (confidence band / market group / band√ógroup cell / exact O/U line / price band, beta-shrunk toward the global rate with `shrink_k` pseudo-counts so thin cells can't dominate); 10 `STRATEGIES` score a tip from its fields and/or those buckets (every scorer total via fallback chains ending at blend confidence ‚Äî pre-2026-07-04 rows lack `tip_breakdown`); `simulateStrategies` replays each strategy per settled day (top-4 slip at real prices, **leave-one-day-out** calibration so calibrated strategies never grade their own answers) and ranks by slip survival ‚Üí pooled top-quarter hit rate ‚Üí roi (user-selected policy, `days ‚â• minDays` tier first); `magicSortRows` (score desc, tipless/skip/AI-vetoed rows sink, stable ties) and `slipSummary`/`estimateLegProb` back the web table and betslip playground. `simulateStrategies` stats include `streak {days, avg, best}` ‚Äî depth-before-first-miss from the top of each day's ranking (display-only; ranking policy unchanged until >30 replay days). **Safe-only selection** (Safety Net Protocol, 2026-07-09): `DEFAULT_SAFE` + `safeQualifies(row)` + `safeSelection(rows, cal, opts)` cherry-pick multi-bet slip legs ‚Äî ‚â•2 blend components present, weakest (`tipAgreement`) ‚â• 0.65, price ‚â§ 1.6, one row per `api_id`, ranked per day by the pinned `market` strategy, top 3/day. Thresholds default to literals (client imports the module verbatim), but the effective policy is overridable via `SAFE_*` env (`SAFE_MAX_PER_DAY`/`SAFE_STRATEGY`/`SAFE_MIN_PARTS`/`SAFE_MIN_AGREEMENT`/`SAFE_MAX_PRICE`) ‚Äî `src/magic.js` reads config and ships the resolved `safe` object in the `/api/magic-sort` payload so the browser (which can't read `.env`) and server can't diverge; `safeSelection(rows, cal, opts)` merges over `DEFAULT_SAFE`. ### Key invariants

- **Fetch throttling:** `matches.completed_at` set ‚áí odds refreshes skip the match. Fixtures reaching a terminal status (`FT/AET/PEN/AWD/WO/CANC/ABD`) complete their linked matches; unlinked matches complete 4h after start (fallback).
- **Fetch-once stats:** `fixtures.stats_fetched_at`/`lineups_fetched_at`/`events_fetched_at` guarantee each final fixture costs at most 3 detail requests ever. Empty responses only set the flag after 48h post-kickoff (minor leagues may never publish stats). Never delete or refetch immutable API data.
- **Pre-match snapshots freeze at kickoff:** `fixture_prematch` rows are upserted every run while the fixture is upcoming and never written after kickoff ‚Äî historical pre-match stats must stay exactly as they were, unaffected by later matches.
- **Hot picks freeze at kickoff and settle exactly once:** `fixture_predictions` rows use the same `kickoff > NOW()` selection-freeze; the settle pass (canonical FT scores ‚Üí `result_goals`/`outcome`) is the only writer of those columns. The hit-rate scoreboard is honest by construction ‚Äî never rewrite a settled or past-kickoff pick. AI adjudication is optional (no `GEMINI_API_KEY` = rules-only) and can only veto, never promote; API `/predictions` uses the fetch-once flag `fixtures.predictions_fetched_at`.
- **Fetch-once history:** `fixtures.history_fetched_at` guarantees each upcoming correlated fixture costs at most 3 backfill requests ever (2√ó team last-N, 1√ó full head-to-head). Only FINAL_STATUSES items are saved ‚Äî headtohead also returns future meetings, which must not leak into the results refresh set.
- **Results are canonical:** the `results` action copies authoritative scores from final fixtures into linked matches (bookmaker-parsed scores are unreliable for upcoming games ‚Äî BetPawa reports 0-0, Betika null).
- **Betika null fields:** `home_team_id, away_team_id, region_id, region_name, category_id, competition_id` are always null (not exposed by its API).
- **Access tiers are server-authoritative (v1.1.0):** guest redaction/date-gating happens in `records.js`/`server.js`, never client-only; the response memo key MUST carry the tier (a guest and a full body can never share a cache slot). `AUTH_ENABLED=0` and machine bearers are the legacy full-access path (`access=null`). Sessions store only sha256 hashes ‚Äî a DB leak yields no usable token; rotating `PIN_PEPPER` invalidates every stored PIN (deliberate global reset, never do it casually).
- Migrations are forward-only; never edit an applied migration.

## Conventions

- ES modules, `async/await`, 4-space indentation (workspace setting ‚Äî overrides the usual 2-space Node default), single quotes, semicolons.
- Foreign keys use `ON DELETE CASCADE`/`RESTRICT`; a NULLABLE audit pointer may use `ON DELETE SET NULL` (prior art: `settings.updated_by`, batch 11 ‚Äî deleting a user must not delete the settings they once touched). Migrations that seed data may read `process.env` directly (they run under the knex CLI as well as the app; prior art: the users migration's `ADMIN_SEED_PIN`/`PIN_PEPPER`).
- All external data (API responses, env) through zod schemas; keep field schemas tolerant (`nullable().optional()`) ‚Äî live data has taught this (`league.round` can be null; `/fixtures/events` `type` can be null ‚Äî parsed by the tolerant `src/apisports-events.js`, and per-fixture ZodErrors are caught so one bad record can't abort a sweep).
- `x-*-output.xx.json` files at the root are legacy fetched-data snapshots ‚Äî do not delete.
