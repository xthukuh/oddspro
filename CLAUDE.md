# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

oddspro is a MySQL data warehouse for football bookmaker odds and stats. It scrapes odds from two Kenyan bookmakers (BetPawa, Betika), ingests canonical fixture/result/stats data from API-Football (api-sports.io), and correlates bookmaker matches to canonical fixtures via fuzzy matching with learned aliases. Plain Node.js (ES modules); knex/mysql2 for the DB layer; zod validates all external data.

Source spec and progress tracking live in `implementation-plan.md` (phases 1–10 done: warehouse, ingestion, linking, deep stats, visualization, stale odds, pre-match snapshots).

## Commands

```sh
npm install
npm run migrate                     # knex migrate:latest (forward-only migrations)

npm run start [-- days]             # DEFAULT full pipeline (src/pipeline.js): fixtures + odds for today..+3 days
                                    # (`npm run start -- 5` overrides the sweep), then results, link once, stats,
                                    # standings, team history, pre-match snapshots
node src/index.js betpawa [date]    # scrape BetPawa odds → DB, then auto-link
node src/index.js betika [date]     # scrape Betika odds → DB, then auto-link
node src/index.js fixtures [date]   # API-Football fixtures for date → DB, then auto-link
node src/index.js results           # refresh unfinished past-kickoff fixtures; settle scores; mark matches completed
node src/index.js link [provider]   # correlate bookmaker matches ↔ canonical fixtures
node src/index.js stats             # statistics + lineups + events for final correlated fixtures (fetch-once)
node src/index.js standings         # refresh league tables for correlated leagues
node src/index.js history           # backfill team last-N + full head-to-head history for upcoming correlated fixtures (fetch-once)
node src/index.js prematch          # upsert pre-match snapshots for upcoming correlated fixtures (frozen once kickoff passes)
node src/index.js predictions       # API-Football /predictions for upcoming correlated fixtures (fetch-once)
node src/index.js hotpicks          # settle + recompute over-2.5 hot picks + tips (rules + optional AI adjudication/review)
node src/index.js performance       # flat-stake ROI / hit-rate / bucket report for tips + hot picks (also GET /api/performance)
node scripts/backtest-hotpicks.js   # replay historical fixtures to measure/tune hot-pick gate precision
node scripts/analyze-safe-tips.js   # weekly: LODO grid of the Safe-only gates + runner-up hypothesis re-tests
                                    # (run BEFORE touching DEFAULT_SAFE in src/db/magic-rules.js)
node src/index.js export [date]     # temp CSV of the date's correlated records → tmp/ (gitignored)

npm run serve                       # visualization API server on :3001 (serves web/dist when built) + the
                                    # in-process auto-refresh scheduler (light pass every AUTO_LIGHT_MINUTES,
                                    # full sweep daily at AUTO_FULL_AT EAT; AUTO_REFRESH_ENABLED=0 opts out locally)
npm run build:web                   # build the React frontend → web/dist/
cd web && npm run dev               # frontend dev server on :5173 (proxies /api/* → :3001)

npm test                            # node:test suite (tests/*.test.js) — offline, no DB/live APIs:
                                    # market registry invariants, stale-odds diff scenarios, pre-match calc
                                    # (H2H/rolling-goals windows), hot-pick rules (gates/devig/veto logic),
                                    # magic-sort rules (calibration/strategies/slip replay, safe-only
                                    # selection gates, streak stat), api-sports rate-limit rules,
                                    # auto-refresh scheduling rules (EAT due-times, log trim) and the client
                                    # freshness reload gate, and the standardized game-record contract
                                    # vs the frozen x-*-output.xx.json snapshots
```

`[date]` defaults to today; accepts anything `new Date()` parses, or `today`/`now`. All actions are idempotent and cron-able — a Windows Task Scheduler task `oddspro-pipeline` runs `scripts/pipeline-task.cmd` (full sweep → `logs/pipeline.log`) daily at 08:00. There is no linter. Restart `npm run serve` after pulling backend changes — a stale server process holds :3001 with old code.

`.env` (gitignored, see `.env.example`) holds MySQL credentials (`DB_HOST/DB_PORT/DB_DATABASE/DB_USERNAME/DB_PASSWORD/DB_CHARSET/DB_COLLATION` — Laravel-style names), API-Football credentials (`X_APISPORTS_URL`, `X_APISPORTS_KEY`), and optionally `GEMINI_API_KEY` (+ `HOTPICK_AI_MODEL` default `gemini-2.5-flash`, `HOTPICK_AI_WEB` — attaches Gemini's `google_search` grounding tool, `TIP_AI_MIN_CONFIDENCE`/`TIP_AI_DAILY_CAP`, `HOTPICK_*`/`TIP_MIN_*` threshold overrides) for the AI adjudicators (Google Gemini; replaced OpenRouter 2026-07-04) — the key lives ONLY in `.env`, never in git. Auto-refresh knobs: `AUTO_REFRESH_ENABLED` (default on), `AUTO_LIGHT_MINUTES` (10), `AUTO_FULL_AT` (`06:00` EAT, `off` disables), `AUTO_FULL_DAYS` (5), `AUTO_LOG`/`AUTO_LOG_MAX_KB` (self-truncating `logs/auto-refresh.log`), `REFRESH_CACHE_MINUTES` (5 — manual refresh answered "fresh" inside the window). Safe-only policy knobs (served to the web via `/api/magic-sort`, effective on server restart, no web rebuild): `SAFE_MAX_PER_DAY` (3), `SAFE_STRATEGY` (`market`), `SAFE_MIN_PARTS` (2), `SAFE_MIN_AGREEMENT` (0.65), `SAFE_MAX_PRICE` (1.6). Deploy/ops knobs (all opt-in, off by default): `MIGRATE_ON_BOOT` (server self-runs `knex migrate:latest` on boot, fail-fast — the no-SSH migration path), and SPA bot-protection — a proof-of-work human gate (`HUMAN_POW_ENABLED` + `VITE_HUMAN_POW` web build, `HUMAN_TOKEN_SECRET`/`HUMAN_POW_BITS`/`HUMAN_TOKEN_TTL_DAYS`/`HUMAN_CHALLENGE_TTL_MINUTES`) plus a bot-UA blocklist + AI `robots.txt` (`BOT_UA_FILTER_ENABLED`/`BOT_UA_EXTRA`/`BOT_UA_ALLOW`); full config in `docs/DEPLOYMENT.md §8`. Validated in `src/config.js` (zod).

## Architecture

Pipeline: **odds scrapers + fixtures ingester → MySQL warehouse → linker correlates → results settle → deep stats accumulate**.

- `src/index.js` — CLI dispatcher; every action closes the shared knex pool on exit. No action (or `start`/a bare day count) runs `src/pipeline.js`.
- `src/pipeline.js` — default full sweep (`npm run start`), ordered for fewest server hits: fixtures per date first (a date fetch also refreshes today's statuses, shrinking the results refresh set), then results (completes matches so scrapers skip their per-game detail requests via the `completedMatchIds()` exclusion set), then odds per provider per date, then a single link pass, then stats + standings + team history + pre-match snapshots. `runStartPipeline(days, onStep)` narrates steps and returns `{ dates, quota_remaining }` (the scheduler's full mode reuses it verbatim). Also exports `runDateRefresh(date, onStep)` — the single-date subset behind the web refresh button (skips results/stats for future dates, history/snapshots for past dates; standings stays owned by the full sweep).
- `src/auto-refresh.js` — in-process scheduler + the single-slot refresh job state shared with the manual endpoint (one guard, auto and manual can never overlap). One unref'd 30s `setInterval` tick: full sweep due once per EAT day at `AUTO_FULL_AT` (stamped at START — one attempt/day even on failure; a mid-day restart never fires a surprise sweep), else light pass every `AUTO_LIGHT_MINUTES` — `lightRefresh()`: `settleApisportsResults()` (statuses/scores/`elapsed` per id, no whole-date fetch) → today's odds per provider (completed excluded) → `linkMatches()` → `settleHotPicks()`. Successful completions bump a monotonic `data_version` + `last_success {at,mode,dates}` (the client freshness signal) and stamp a success-only per-date freshness map (`lastFreshAt`) — deliberately separate from server.js's manual cooldown map (a 10-min cadence stamping the cooldown would keep today permanently 429'd). Per-job lines land in self-truncating `logs/auto-refresh.log` (`AUTO_LOG_MAX_KB`). Pure schedule math (EAT due-times, `trimLogTail`) lives in zero-import `src/db/auto-rules.js` (offline-tested).
- `src/betpawa.js` / `src/betika.js` — bookmaker scrapers (browser-mimicking axios clients against undocumented public APIs). Both emit the same standardized game record consumed by `store.saveMatches()`.
- `src/apisports.js` — API-Football client + all its fetchers. Quota-guarded (`x-ratelimit-requests-remaining` header, halts at `APISPORTS_MIN_REMAINING` floor), paginated, zod-validated. The per-minute burst budget (`x-ratelimit-remaining` header) is paced, not fatal: `_getPage` sleeps into the next minute window when it runs out and retries an `errors.rateLimit` response (bounded, 2 max) — decision logic in pure `src/db/rate-rules.js` (offline-tested; the daily-quota error deliberately does NOT match and stays fatal). Fixtures are fetched with `timezone=Africa/Nairobi` so kickoffs align with bookmaker wall-clock times.
- `src/link.js` — correlation. API-Football fixtures are the **canonical base record**; `matches.fixture_id` is the link. Provider order matters: betpawa first, betika last (betika lacks identifier attributes and additionally scores against betpawa matches already linked to a candidate fixture). Fuzzy scorer = best of bigram-Dice / token-set / overlap-coefficient / initialism over normalized names; competition similarity is a bonus (+0.1×sim), never a veto; acceptance needs `LINK_MIN_CONFIDENCE` (default 0.85) plus a 0.05 margin over the runner-up. Confident links cache `team_aliases` / `league_aliases` for instant future correlation.
- `src/db/connection.js` — the only knex instance (never use raw mysql2). Session `time_zone` is pinned to +03:00 in `knexfile.js` so SQL `NOW()` compares correctly against stored EAT wall-clock datetimes.
- `src/db/store.js` — odds persistence: upsert `matches` by `(provider, provider_match_id)` (with an explicit `updated_at` bump — MySQL's ON UPDATE skips no-op updates), then refresh `odds_markets` via `src/db/odds-diff.js`: markets present in the latest snapshot are replaced (delete+insert), vanished ones are kept flagged `is_stale` (last-seen price survives for display), re-listed markets revive. Never touches `completed_at`/`fixture_id` (owned by results/link).
- `src/prematch.js` — pre-match snapshot writer (`updatePrematchSnapshots()`): upserts `fixture_prematch` (rank/form/H2H/rolling-goals aggregates) for every upcoming correlated fixture; the `kickoff > NOW()` selection **is** the freeze — past fixtures are never selected again, so their last pre-kickoff snapshot stands forever. Single-statement chunked upserts (no delete+insert, no deadlock exposure).
- `src/hotpicks.js` — over-2.5 hot picks + best tips (`updateHotPicks()`): settles `fixture_predictions` outcomes via the standalone `settleHotPicks()` export (hot picks via SQL, tips via pure `tipHit` — cheap, no fetches/AI, also called by the light auto-refresh pass) from canonical final scores, then re-evaluates every upcoming correlated snapshot-backed fixture through the deduction gates; rule-hot candidates optionally pass a Gemini confirm/veto (`src/ai.js`, fail-open — errors keep the rule verdict; verdicts are reused while the score AND the `aiModelTag()` are unchanged, so reruns don't re-bill and model/grounding/prompt-version switches re-adjudicate automatically — the tag carries a `#p<N>` prompt-version suffix). Adjudicator prompts are v2 (2026-07-05, de-anchored): the model researches context + team news (google_search when `HOTPICK_AI_WEB=1`), states its OWN probability, and vetoes mechanically only below break-even − 0.05 or on a verified disqualifier — market-vs-stats contradiction alone is never a veto ground (v1's contradiction vetoes were net-negative). Structured verdicts (probability, per-check findings, grounding citations) persist to `ai_review`/`tip_ai_review` JSON; reply decoding lives in pure `src/ai-parse.js` (zod-only, offline-testable). Tips run `tipEligibility` first (thin-evidence fixtures store `tip_skip_reason` instead of a market-only tip), persist the full `bestTip` blend as `tip_breakdown` JSON, and the top-confidence tips (`TIP_AI_MIN_CONFIDENCE`, capped `TIP_AI_DAILY_CAP`/run) get an AI review (`reviewTip`, same reuse idiom keyed on tip market+price) — a tip veto FLAGS the row (`tip_ai_verdict`) but never clears it, so its settled outcome measures what the veto was worth. Same freeze idiom as prematch: `kickoff > NOW()` selection, chunked onConflict-merge upsert (merge list excludes `result_goals`/`outcome`/`tip_outcome` — owned by the settle pass). `hotpicksSummary()` backs `GET /api/hotpicks`; `performanceSummary()` (thin loader over pure `src/db/perf-rules.js`) backs `GET /api/performance` + the `performance` CLI report.
- `src/db/goals-rules.js` — pure hot-pick deduction rules (zero imports, offline-testable): vig-removed `impliedProbability`, kickoff-cutoff team/H2H goal aggregates, API-Football prediction signal parsing, and `scoreOver25` — strict AND concurrence gates (both teams' rolling avg-total + over-rate, market probability floor, H2H veto, API veto), precision over recall by design. **Fairness pairing** (`pairedTeamGoalsAggregates`, mirrored by tip-rules' `pairedTeamOutcomeAggregates`): both teams of a fixture are judged over the SAME window, capped at the smaller side's qualifying count — mixed windows bias the rate gates; `pool` keeps the uncapped counts for eligibility attribution. `DEFAULT_THRESHOLDS` tuned by `scripts/backtest-hotpicks.js` (54.3% baseline → 73.3% stats-only precision over 10,678 fixtures, re-validated under pairing 2026-07-04); `HOTPICK_TEAM_WINDOW` must stay ≤ ~8 (history backfill only fetches 10 games/team).
- `src/db/tip-rules.js` — pure "Tip" column deduction (zero imports): `teamOutcomeAggregates`/`h2hOutcomeAggregates` (W/D/L + per-line over rates, kickoff cutoff), `tipEligibility` — evidence screen run BEFORE `bestTip` (context gate FIRST: `TIP_CONTEXT_EXCLUDE` skips friendly/youth/reserve leagues outright — rolling form is invalid evidence there, the epicenter of the first settled losses; then both teams' sample ≥ `minGames` + at least one full market group; a market-only blend is just the bookmaker's own devigged opinion and can't beat the vig), `bestTip` — safest bettable outcome across 1X2/double-chance/every O/U line, confidence = renormalized blend of devigged market probability (0.6) + rolling-stats support (0.3) + API-Football percentages (0.1), with a price floor (`TIP_MIN_PRICE`, default 1.2) that excludes near-certain junk odds — what survives at high confidence is the "hidden gem" — and an Under line floor (`TIP_MIN_UNDER_LINE`, default 4.5: no U 2.5/U 3.5 tips — near-Unders realized 61.9% vs 78.1% break-even; suppressed Unders yield to the runner-up); the return carries the justification breakdown (component probs, effective weights, samples, runners-up) persisted verbatim as `tip_breakdown`. `tipHit` settles any canonical market from a final score. Tips live on `fixture_predictions` (`tip_market/tip_price/tip_confidence/tip_outcome` + `tip_breakdown/tip_skip_reason/tip_ai_*`), freeze at kickoff, sort/filter via the `tip` base column (confidence).
- `src/db/perf-rules.js` — pure betting-performance calc (zero imports): `summarizePerformance(rows)` → flat-stake ROI (1 unit/settled pick), hit-rate, avg price + break-even rate per 7d/30d/all window, buckets by confidence band / market group / exact O/U line (tips only — keeps the near-Under and tail-Over residuals measurable) / edge sign (`confidence × price − 1`, the EV proxy), and AI-veto impact (vetoed picks settle but are excluded from headlines — `saved` measures what following the vetoes was worth). Hit-rate alone can't prove profit; the edge buckets are where false positives are located.
- `src/db/magic-rules.js` — pure magic-sort engine (imports only the perf-rules bucket labelers, still offline-testable) shared VERBATIM by the server loader and the web table (vite `server.fs.allow ['..']` serves the out-of-root import in dev; the build follows relative imports natively — one scorer, no client/server drift): `computeCalibration` digests every settled tip into empirical hit-rate buckets (confidence band / market group / band×group cell / exact O/U line / price band, beta-shrunk toward the global rate with `shrink_k` pseudo-counts so thin cells can't dominate); 10 `STRATEGIES` score a tip from its fields and/or those buckets (every scorer total via fallback chains ending at blend confidence — pre-2026-07-04 rows lack `tip_breakdown`); `simulateStrategies` replays each strategy per settled day (top-4 slip at real prices, **leave-one-day-out** calibration so calibrated strategies never grade their own answers) and ranks by slip survival → pooled top-quarter hit rate → roi (user-selected policy, `days ≥ minDays` tier first); `magicSortRows` (score desc, tipless/skip/AI-vetoed rows sink, stable ties) and `slipSummary`/`estimateLegProb` back the web table and betslip playground. `simulateStrategies` stats include `streak {days, avg, best}` — depth-before-first-miss from the top of each day's ranking (display-only; ranking policy unchanged until >30 replay days). **Safe-only selection** (Safety Net Protocol, 2026-07-09): `DEFAULT_SAFE` + `safeQualifies(row)` + `safeSelection(rows, cal, opts)` cherry-pick multi-bet slip legs — ≥2 blend components present, weakest (`tipAgreement`) ≥ 0.65, price ≤ 1.6, one row per `api_id`, ranked per day by the pinned `market` strategy, top 3/day. Thresholds default to literals (client imports the module verbatim), but the effective policy is overridable via `SAFE_*` env (`SAFE_MAX_PER_DAY`/`SAFE_STRATEGY`/`SAFE_MIN_PARTS`/`SAFE_MIN_AGREEMENT`/`SAFE_MAX_PRICE`) — `src/magic.js` reads config and ships the resolved `safe` object in the `/api/magic-sort` payload so the browser (which can't read `.env`) and server can't diverge; `safeSelection(rows, cal, opts)` merges over `DEFAULT_SAFE`. Re-tune the gate thresholds via `scripts/analyze-safe-tips.js` (LODO grid; the 2026-07-09 run replayed 94.4% legs at 2.6 picks/day). The runner-up swap was backtested net-negative (+108/−128) — `bestTip` stays untouched, the script re-tests the hypothesis weekly. Calibrated strategies are the "gets better as data grows" part — live data shows raw confidence is NOT monotonic with winning.
- `src/magic.js` — thin knex loader behind `GET /api/magic-sort`: `settledTipRows()` (settled tips + `DATE_FORMAT(kickoff)` day — grouping stays in the pinned +03:00 SQL session — plus the final-score pair for the analysis script; shared with `scripts/analyze-safe-tips.js` so the two never drift), `simulateStrategies`, and the `safe` policy object (SAFE_* env → the client's toggle) in the payload; per-day in-memory cache (`?refresh=1` recomputes; failed compute clears the slot).
- `src/db/prematch-calc.js` — pure calculations shared by the snapshot writer and the read-layer live fallback (`h2hSummary`, `computePrematch`, `formatGoals`; zero imports so tests skip config/.env). Callers filter `status IN RESULT_STATUSES` in SQL; the calc enforces non-null FT scores + kickoff strictly before the fixture's kickoff. Rolling windows configurable via `PREMATCH_TEAM_WINDOW` / `PREMATCH_H2H_WINDOW` (default 5/5).
- `src/db/odds-diff.js` — pure diff helper (`oddsIdentity`, `diffOddsRows`; zero imports so tests skip config/.env). Identity = type_name/name/handicap with numeric handicap normalization — mysql2 returns DECIMAL as a string (`'2.5'`) while snapshots carry numbers.
- `src/db/filter-csv.js` — pure CSV-list parser (`parseFilterList`; zero imports, offline-tested) for the `in`/`not-in` filter operators, shared VERBATIM by the server (`records.js`) and web (`filterValues.js`, out-of-root import). Comma-separated; double-quote an item to include commas/spaces (`""` = literal quote); empty items dropped.
- `src/utils.js` — shared helpers (`_date`, `_dtime`, `_batch`). `_batch` runs promises with bounded concurrency; keep DB-writing batches at concurrency 1 — parallel delete+insert transactions deadlock on InnoDB index gap locks.
- `src/markets.js` — canonical odds market columns (1, X, 2, 1X, X2, 12, U/O 0.5–6.5; defaults 1.5–4.5). Single registry drives both the JS odds pivot and the SQL sort/filter conditions. Match markets by `type_name`, never `type_id` (betika reuses ids across different markets).
- `src/db/records.js` — read layer for visualization: `queryRecords()` (correlated records only; multi-sort, filters — ops `eq/ne/gt/gte/lt/lte`, text `like`/`not-contains`, CSV-list `in`/`not-in` (via `filter-csv.js`); text/set ops on a base field with a `like_sql` target, e.g. `tip`, match its text (`fp.tip_market`) — optional pagination — `per_page: 'all'` disables it (the web table), the CSV export still pages; `providers` array filters bookmakers; `completed: false` hides concluded games) and `columnCatalog()` (STATS columns discovered dynamically from `fixture_statistics`; `providers` discovered from `matches` so new bookmakers appear in the settings UI without frontend changes). Pre-match stats prefer the frozen `fixture_prematch` snapshot when one exists (presence of the row wins wholesale — a null snapshot rank means "no rank at kickoff" and must not drift back to live standings); fixtures predating the feature fall back to live rank/form (standings) and H2H (fixtures history) derivation. The rolling-goals columns (`h2h_count`, `home/away_goals_h2h`, `home/away_goals_oth`, compact `"gf/ga (avg)"` strings) are snapshot-only. Score/goals are only surfaced once the fixture status is final (bookmaker pre-match scores are garbage). Rows also carry `updated_at` (odds refresh time), `markets_stale` (last-seen prices of vanished markets), `available` (false once the fixture is terminal, the match completed, or the latest update had no markets) and `elapsed` (live match minute, display-only — the Status tooltip appends it for in-play statuses; refreshed by the results settle pass, so ≤ one light cadence stale).
- `src/export.js` / `src/server.js` — CSV export action and the :3001 Express API (`GET /api/records`, `GET /api/columns`; `sort`/`filters` are JSON-encoded query params validated against the column registries — unknown keys/ops are a 400 (ops `eq/ne/gt/gte/lt/lte/like/not-contains/in/not-in`; `in`/`not-in` take a CSV-list value); `completed=0` hides concluded games, `providers=a,b` filters bookmakers, `per_page=all` returns the whole selection — the web table is unpaginated; `GET /api/magic-sort` serves the backtest-ranked tip strategies + calibration, see `src/magic.js`). `POST /api/refresh?date=` starts a single-slot background `runDateRefresh` job — the slot lives in `src/auto-refresh.js` and is shared with the scheduler (409 while ANY job runs — parallel refreshes would deadlock on delete+insert gap locks; `200 {fresh:true}` when the date was successfully refreshed within `REFRESH_CACHE_MINUTES`; 429 manual cooldown). `GET /api/refresh` returns the job state + freshness signal (`data_version`, `last_success`) — the web app polls it slowly (60s) for silent reloads and fast (2s) while a job runs. The POST requires an `X-Requested-With` header (CSRF guard — custom headers force a CORS preflight this server never approves), and the server binds `API_HOST` (default `127.0.0.1`; set `0.0.0.0` to expose on the LAN). The server starts the scheduler after listen and stops it on shutdown; when `MIGRATE_ON_BOOT` is set it awaits `db.migrate.latest()` BEFORE listening (fail-fast on error — summary via pure `src/db/migrate-rules.js`). Opt-in bot-protection (all off by default; pure logic in `src/human-pow.js` + `src/bot-rules.js`, offline-tested): a `BOT_UA_FILTER_ENABLED` middleware 403s known bot/AI-scraper user-agents site-wide (`isBlockedUserAgent`; Googlebot/Bingbot exempt), `GET /robots.txt` always disallows AI crawlers + `/api/`, and when `HUMAN_POW_ENABLED` a stateless proof-of-work gate (`GET /api/challenge` → browser solves → `POST /api/human` mints a check-once HMAC token) makes every other `/api/*` require an `X-Human-Token` (401 without it; an `API_TOKEN` bearer bypasses). The refresh writers are deadlock-retried (`src/db/retry-rules.js` `withRetry` wraps `apisports._saveFixtureItems` + `store.saveMatches` — transient InnoDB deadlock/lock-timeout self-heals; run only ONE `serve`, a second concurrent writer is the deadlock source).
- `web/` — React 19 + Vite 6 + Tailwind 4 datatable (compact `text-xs`; **iPadOS-native look + light/dark, Phase 23**). **Theme:** a Tailwind 4 `@theme inline` + CSS-variable token layer in `web/src/index.css` (`--surface/--label/--accent/--separator/--fill/--nav-bg/--logo/--green(hit)/--orange(hot)/--red(miss)` …) flips light/dark on `prefers-color-scheme` and a `:root[data-theme]` override (wins both ways); font stack `-apple-system,…,Inter Variable` (SF on iPad, self-hosted Inter fallback). Shared iOS primitives: `Sheet.jsx` (centered sheet, blur scrim, Escape/backdrop dismiss), `Logo.jsx` (theme-adaptive `[OP]` mark, home→today), `CalendarPopover.jsx`, `OverflowMenu.jsx`, `icons.jsx` (SVG line icons + spinner). The whole page is a fixed `100dvh` app shell — only the table scrolls. **Presentation-only: Score/Tip pins, sort, scroll-preservation, magic column, filters and all behavior are unchanged; suite stayed 185/185.** Row tints cycle per canonical fixture (`api_id`) so the same match shown per provider shares a color; row tooltips surface the odds refresh time; the Status cell tooltip appends the live minute for in-play games (`Second half (live) — 67'`, from `fixtures.elapsed`); stale market prices render greyed, and a frozen match (unavailable — concluded / no live markets) greys ALL its prices the same way; unavailable matches lose their link unless re-enabled per provider in Settings. The table is unpaginated (whole selected date at once, record count below); missed tips render red, AI-vetoed tips struck through, and clicking a tip cell opens the justification popover (`TipPopover.jsx`, phrased for laymen: "Why this tip" blend with signals named in plain words (bookmaker odds / recent form / expert data) and weights as %-of-verdict, evidence samples, runner-up candidates ("Close alternatives"), the over-2.5 gate audit labeled via the exported `SIGNAL_LABEL` glossary (reused by the table's 🔥 badge tooltip — one glossary, no drift), spelled-out AI verdicts, and a 🛡 Safe-pick badge on rows passing `safeQualifies` — v2 reviews additionally render the model's own probability, per-check findings and grounding source links; ineligible fixtures show a muted "no data" marker backed by `tip_skip_reason`). ### Key invariants

- **Fetch throttling:** `matches.completed_at` set ⇒ odds refreshes skip the match. Fixtures reaching a terminal status (`FT/AET/PEN/AWD/WO/CANC/ABD`) complete their linked matches; unlinked matches complete 4h after start (fallback).
- **Fetch-once stats:** `fixtures.stats_fetched_at`/`lineups_fetched_at`/`events_fetched_at` guarantee each final fixture costs at most 3 detail requests ever. Empty responses only set the flag after 48h post-kickoff (minor leagues may never publish stats). Never delete or refetch immutable API data.
- **Pre-match snapshots freeze at kickoff:** `fixture_prematch` rows are upserted every run while the fixture is upcoming and never written after kickoff — historical pre-match stats must stay exactly as they were, unaffected by later matches.
- **Hot picks freeze at kickoff and settle exactly once:** `fixture_predictions` rows use the same `kickoff > NOW()` selection-freeze; the settle pass (canonical FT scores → `result_goals`/`outcome`) is the only writer of those columns. The hit-rate scoreboard is honest by construction — never rewrite a settled or past-kickoff pick. AI adjudication is optional (no `GEMINI_API_KEY` = rules-only) and can only veto, never promote; API `/predictions` uses the fetch-once flag `fixtures.predictions_fetched_at`.
- **Fetch-once history:** `fixtures.history_fetched_at` guarantees each upcoming correlated fixture costs at most 3 backfill requests ever (2× team last-N, 1× full head-to-head). Only FINAL_STATUSES items are saved — headtohead also returns future meetings, which must not leak into the results refresh set.
- **Results are canonical:** the `results` action copies authoritative scores from final fixtures into linked matches (bookmaker-parsed scores are unreliable for upcoming games — BetPawa reports 0-0, Betika null).
- **Betika null fields:** `home_team_id, away_team_id, region_id, region_name, category_id, competition_id` are always null (not exposed by its API).
- Migrations are forward-only; never edit an applied migration.

## Conventions

- ES modules, `async/await`, 4-space indentation (workspace setting — overrides the usual 2-space Node default), single quotes, semicolons.
- All external data (API responses, env) through zod schemas; keep field schemas tolerant (`nullable().optional()`) — live data has taught this (`league.round` can be null).
- `x-*-output.xx.json` files at the root are legacy fetched-data snapshots — do not delete.
