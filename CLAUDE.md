## Project Overview

oddspro is a MySQL data warehouse for football bookmaker odds and stats. It scrapes odds from two Kenyan bookmakers (BetPawa, Betika), ingests canonical fixture/result/stats data from API-Football (api-sports.io), and correlates bookmaker matches to canonical fixtures via fuzzy matching with learned aliases. Plain Node.js (ES modules); knex/mysql2 for the DB layer; zod validates all external data.

Source spec and progress tracking live in `implementation-plan.md` (phases 1â€“10 done: warehouse, ingestion, linking, deep stats, visualization, stale odds, pre-match snapshots).

node src/index.js standings         # refresh league tables for correlated leagues
node src/index.js history           # backfill team last-N + full head-to-head history for upcoming correlated fixtures (fetch-once)

node src/index.js predictions       # API-Football /predictions for upcoming correlated fixtures (fetch-once)
node src/index.js hotpicks          # settle + recompute over-2.5 hot picks + tips (rules + optional AI adjudication/review)

node scripts/backtest-hotpicks.js   # replay historical fixtures to measure/tune hot-pick gate precision
node scripts/analyze-safe-tips.js   # weekly: LODO grid of the Safe-only gates + runner-up hypothesis re-tests

npm run serve                       # visualization API server on :3001 (serves web/dist when built) + the
                                    # in-process auto-refresh scheduler (light pass every AUTO_LIGHT_MINUTES,
                                    # full sweep daily at AUTO_FULL_AT EAT; AUTO_REFRESH_ENABLED=0 opts out locally)
npm run build:web                   # build the React frontend â†’ web/dist/
cd web && npm run dev               # frontend dev server on :5173 (proxies /api/* â†’ :3001)

                                    # market registry invariants, stale-odds diff scenarios, pre-match calc

                                    # magic-sort rules (calibration/strategies/slip replay, safe-only

                                    # auto-refresh scheduling rules (EAT due-times, log trim) and the client

                                    # vs the frozen x-*-output.xx.json snapshots

`[date]` defaults to today; accepts anything `new Date()` parses, or `today`/`now`. All actions are idempotent and cron-able â€” a Windows Task Scheduler task `oddspro-pipeline` runs `scripts/pipeline-task.cmd` (full sweep â†’ `logs/pipeline.log`) daily at 08:00. There is no linter. Restart `npm run serve` after pulling backend changes â€” a stale server process holds :3001 with old code.

`.env` (gitignored, see `.env.example`) holds MySQL credentials (`DB_HOST/DB_PORT/DB_DATABASE/DB_USERNAME/DB_PASSWORD/DB_CHARSET/DB_COLLATION` â€” Laravel-style names), API-Football credentials (`X_APISPORTS_URL`, `X_APISPORTS_KEY`), and optionally `GEMINI_API_KEY` (+ `HOTPICK_AI_MODEL` default `gemini-2.5-flash`, `HOTPICK_AI_WEB` â€” attaches Gemini's `google_search` grounding tool, `TIP_AI_MIN_CONFIDENCE`/`TIP_AI_DAILY_CAP`, `HOTPICK_*`/`TIP_MIN_*` threshold overrides) for the AI adjudicators (Google Gemini; replaced OpenRouter 2026-07-04) â€” the key lives ONLY in `.env`, never in git. Auto-refresh knobs: `AUTO_REFRESH_ENABLED` (default on), `AUTO_LIGHT_MINUTES` (10), `AUTO_FULL_AT` (`06:00` EAT, `off` disables), `AUTO_FULL_DAYS` (5), `AUTO_LOG`/`AUTO_LOG_MAX_KB` (self-truncating `logs/auto-refresh.log`), `REFRESH_CACHE_MINUTES` (5 â€” manual refresh answered "fresh" inside the window). Safe-only policy knobs (served to the web via `/api/magic-sort`, effective on server restart, no web rebuild): `SAFE_MAX_PER_DAY` (3), `SAFE_STRATEGY` (`market`), `SAFE_MIN_PARTS` (2), `SAFE_MIN_AGREEMENT` (0.65), `SAFE_MAX_PRICE` (1.6). Validated in `src/config.js` (zod).

## Architecture

Pipeline: **odds scrapers + fixtures ingester â†’ MySQL warehouse â†’ linker correlates â†’ results settle â†’ deep stats accumulate**.

- `src/index.js` â€” CLI dispatcher; every action closes the shared knex pool on exit. No action (or `start`/a bare day count) runs `src/pipeline.js`.
- `src/pipeline.js` â€” default full sweep (`npm run start`), ordered for fewest server hits: fixtures per date first (a date fetch also refreshes today's statuses, shrinking the results refresh set), then results (completes matches so scrapers skip their per-game detail requests via the `completedMatchIds()` exclusion set), then odds per provider per date, then a single link pass, then stats + standings + team history + pre-match snapshots. `runStartPipeline(days, onStep)` narrates steps and returns `{ dates, quota_remaining }` (the scheduler's full mode reuses it verbatim). Also exports `runDateRefresh(date, onStep)` â€” the single-date subset behind the web refresh button (skips results/stats for future dates, history/snapshots for past dates; standings stays owned by the full sweep).
- `src/auto-refresh.js` â€” in-process scheduler + the single-slot refresh job state shared with the manual endpoint (one guard, auto and manual can never overlap). One unref'd 30s `setInterval` tick: full sweep due once per EAT day at `AUTO_FULL_AT` (stamped at START â€” one attempt/day even on failure; a mid-day restart never fires a surprise sweep), else light pass every `AUTO_LIGHT_MINUTES` â€” `lightRefresh()`: `settleApisportsResults()` (statuses/scores/`elapsed` per id, no whole-date fetch) â†’ today's odds per provider (completed excluded) â†’ `linkMatches()` â†’ `settleHotPicks()`. Successful completions bump a monotonic `data_version` + `last_success {at,mode,dates}` (the client freshness signal) and stamp a success-only per-date freshness map (`lastFreshAt`) â€” deliberately separate from server.js's manual cooldown map (a 10-min cadence stamping the cooldown would keep today permanently 429'd). Per-job lines land in self-truncating `logs/auto-refresh.log` (`AUTO_LOG_MAX_KB`). Pure schedule math (EAT due-times, `trimLogTail`) lives in zero-import `src/db/auto-rules.js` (offline-tested).
- `src/betpawa.js` / `src/betika.js` â€” bookmaker scrapers (browser-mimicking axios clients against undocumented public APIs). Both emit the same standardized game record consumed by `store.saveMatches()`.
- `src/apisports.js` â€” API-Football client + all its fetchers. Quota-guarded (`x-ratelimit-requests-remaining` header, halts at `APISPORTS_MIN_REMAINING` floor), paginated, zod-validated. The per-minute burst budget (`x-ratelimit-remaining` header) is paced, not fatal: `_getPage` sleeps into the next minute window when it runs out and retries an `errors.rateLimit` response (bounded, 2 max) â€” decision logic in pure `src/db/rate-rules.js` (offline-tested; the daily-quota error deliberately does NOT match and stays fatal). Fixtures are fetched with `timezone=Africa/Nairobi` so kickoffs align with bookmaker wall-clock times.
- `src/link.js` â€” correlation. API-Football fixtures are the **canonical base record**; `matches.fixture_id` is the link. Provider order matters: betpawa first, betika last (betika lacks identifier attributes and additionally scores against betpawa matches already linked to a candidate fixture). Fuzzy scorer = best of bigram-Dice / token-set / overlap-coefficient / initialism over normalized names; competition similarity is a bonus (+0.1Ã—sim), never a veto; acceptance needs `LINK_MIN_CONFIDENCE` (default 0.85) plus a 0.05 margin over the runner-up. Confident links cache `team_aliases` / `league_aliases` for instant future correlation.
- `src/db/connection.js` â€” the only knex instance (never use raw mysql2). Session `time_zone` is pinned to +03:00 in `knexfile.js` so SQL `NOW()` compares correctly against stored EAT wall-clock datetimes.
- `src/db/store.js` â€” odds persistence: upsert `matches` by `(provider, provider_match_id)` (with an explicit `updated_at` bump â€” MySQL's ON UPDATE skips no-op updates), then refresh `odds_markets` via `src/db/odds-diff.js`: markets present in the latest snapshot are replaced (delete+insert), vanished ones are kept flagged `is_stale` (last-seen price survives for display), re-listed markets revive. Never touches `completed_at`/`fixture_id` (owned by results/link).
- `src/prematch.js` â€” pre-match snapshot writer (`updatePrematchSnapshots()`): upserts `fixture_prematch` (rank/form/H2H/rolling-goals aggregates) for every upcoming correlated fixture; the `kickoff > NOW()` selection **is** the freeze â€” past fixtures are never selected again, so their last pre-kickoff snapshot stands forever. Single-statement chunked upserts (no delete+insert, no deadlock exposure).
- `src/hotpicks.js` â€” over-2.5 hot picks + best tips (`updateHotPicks()`): settles `fixture_predictions` outcomes via the standalone `settleHotPicks()` export (hot picks via SQL, tips via pure `tipHit` â€” cheap, no fetches/AI, also called by the light auto-refresh pass) from canonical final scores, then re-evaluates every upcoming correlated snapshot-backed fixture through the deduction gates; rule-hot candidates optionally pass a Gemini confirm/veto (`src/ai.js`, fail-open â€” errors keep the rule verdict; verdicts are reused while the score AND the `aiModelTag()` are unchanged, so reruns don't re-bill and model/grounding/prompt-version switches re-adjudicate automatically â€” the tag carries a `#p<N>` prompt-version suffix). Adjudicator prompts are v2 (2026-07-05, de-anchored): the model researches context + team news (google_search when `HOTPICK_AI_WEB=1`), states its OWN probability, and vetoes mechanically only below break-even âˆ’ 0.05 or on a verified disqualifier â€” market-vs-stats contradiction alone is never a veto ground (v1's contradiction vetoes were net-negative). Structured verdicts (probability, per-check findings, grounding citations) persist to `ai_review`/`tip_ai_review` JSON; reply decoding lives in pure `src/ai-parse.js` (zod-only, offline-testable). Tips run `tipEligibility` first (thin-evidence fixtures store `tip_skip_reason` instead of a market-only tip), persist the full `bestTip` blend as `tip_breakdown` JSON, and the top-confidence tips (`TIP_AI_MIN_CONFIDENCE`, capped `TIP_AI_DAILY_CAP`/run) get an AI review (`reviewTip`, same reuse idiom keyed on tip market+price) â€” a tip veto FLAGS the row (`tip_ai_verdict`) but never clears it, so its settled outcome measures what the veto was worth. Same freeze idiom as prematch: `kickoff > NOW()` selection, chunked onConflict-merge upsert (merge list excludes `result_goals`/`outcome`/`tip_outcome` â€” owned by the settle pass). `hotpicksSummary()` backs `GET /api/hotpicks`; `performanceSummary()` (thin loader over pure `src/db/perf-rules.js`) backs `GET /api/performance` + the `performance` CLI report.
- `src/db/goals-rules.js` â€” pure hot-pick deduction rules (zero imports, offline-testable): vig-removed `impliedProbability`, kickoff-cutoff team/H2H goal aggregates, API-Football prediction signal parsing, and `scoreOver25` â€” strict AND concurrence gates (both teams' rolling avg-total + over-rate, market probability floor, H2H veto, API veto), precision over recall by design. **Fairness pairing** (`pairedTeamGoalsAggregates`, mirrored by tip-rules' `pairedTeamOutcomeAggregates`): both teams of a fixture are judged over the SAME window, capped at the smaller side's qualifying count â€” mixed windows bias the rate gates; `pool` keeps the uncapped counts for eligibility attribution. `DEFAULT_THRESHOLDS` tuned by `scripts/backtest-hotpicks.js` (54.3% baseline â†’ 73.3% stats-only precision over 10,678 fixtures, re-validated under pairing 2026-07-04); `HOTPICK_TEAM_WINDOW` must stay â‰¤ ~8 (history backfill only fetches 10 games/team).
- `src/db/tip-rules.js` â€” pure "Tip" column deduction (zero imports): `teamOutcomeAggregates`/`h2hOutcomeAggregates` (W/D/L + per-line over rates, kickoff cutoff), `tipEligibility` â€” evidence screen run BEFORE `bestTip` (context gate FIRST: `TIP_CONTEXT_EXCLUDE` skips friendly/youth/reserve leagues outright â€” rolling form is invalid evidence there, the epicenter of the first settled losses; then both teams' sample â‰¥ `minGames` + at least one full market group; a market-only blend is just the bookmaker's own devigged opinion and can't beat the vig), `bestTip` â€” safest bettable outcome across 1X2/double-chance/every O/U line, confidence = renormalized blend of devigged market probability (0.6) + rolling-stats support (0.3) + API-Football percentages (0.1), with a price floor (`TIP_MIN_PRICE`, default 1.2) that excludes near-certain junk odds â€” what survives at high confidence is the "hidden gem" â€” and an Under line floor (`TIP_MIN_UNDER_LINE`, default 4.5: no U 2.5/U 3.5 tips â€” near-Unders realized 61.9% vs 78.1% break-even; suppressed Unders yield to the runner-up); the return carries the justification breakdown (component probs, effective weights, samples, runners-up) persisted verbatim as `tip_breakdown`. `tipHit` settles any canonical market from a final score. Tips live on `fixture_predictions` (`tip_market/tip_price/tip_confidence/tip_outcome` + `tip_breakdown/tip_skip_reason/tip_ai_*`), freeze at kickoff, sort/filter via the `tip` base column (confidence).
- `src/db/perf-rules.js` â€” pure betting-performance calc (zero imports): `summarizePerformance(rows)` â†’ flat-stake ROI (1 unit/settled pick), hit-rate, avg price + break-even rate per 7d/30d/all window, buckets by confidence band / market group / exact O/U line (tips only â€” keeps the near-Under and tail-Over residuals measurable) / edge sign (`confidence Ã— price âˆ’ 1`, the EV proxy), and AI-veto impact (vetoed picks settle but are excluded from headlines â€” `saved` measures what following the vetoes was worth). Hit-rate alone can't prove profit; the edge buckets are where false positives are located.
- `src/db/magic-rules.js` â€” pure magic-sort engine (imports only the perf-rules bucket labelers, still offline-testable) shared VERBATIM by the server loader and the web table (vite `server.fs.allow ['..']` serves the out-of-root import in dev; the build follows relative imports natively â€” one scorer, no client/server drift): `computeCalibration` digests every settled tip into empirical hit-rate buckets (confidence band / market group / bandÃ—group cell / exact O/U line / price band, beta-shrunk toward the global rate with `shrink_k` pseudo-counts so thin cells can't dominate); 10 `STRATEGIES` score a tip from its fields and/or those buckets (every scorer total via fallback chains ending at blend confidence â€” pre-2026-07-04 rows lack `tip_breakdown`); `simulateStrategies` replays each strategy per settled day (top-4 slip at real prices, **leave-one-day-out** calibration so calibrated strategies never grade their own answers) and ranks by slip survival â†’ pooled top-quarter hit rate â†’ roi (user-selected policy, `days â‰¥ minDays` tier first); `magicSortRows` (score desc, tipless/skip/AI-vetoed rows sink, stable ties) and `slipSummary`/`estimateLegProb` back the web table and betslip playground. `simulateStrategies` stats include `streak {days, avg, best}` â€” depth-before-first-miss from the top of each day's ranking (display-only; ranking policy unchanged until >30 replay days). **Safe-only selection** (Safety Net Protocol, 2026-07-09): `DEFAULT_SAFE` + `safeQualifies(row)` + `safeSelection(rows, cal, opts)` cherry-pick multi-bet slip legs â€” â‰¥2 blend components present, weakest (`tipAgreement`) â‰¥ 0.65, price â‰¤ 1.6, one row per `api_id`, ranked per day by the pinned `market` strategy, top 3/day. ### Key invariants

- **Fetch throttling:** `matches.completed_at` set â‡’ odds refreshes skip the match. Fixtures reaching a terminal status (`FT/AET/PEN/AWD/WO/CANC/ABD`) complete their linked matches; unlinked matches complete 4h after start (fallback).
- **Fetch-once stats:** `fixtures.stats_fetched_at`/`lineups_fetched_at`/`events_fetched_at` guarantee each final fixture costs at most 3 detail requests ever. Empty responses only set the flag after 48h post-kickoff (minor leagues may never publish stats). Never delete or refetch immutable API data.
- **Pre-match snapshots freeze at kickoff:** `fixture_prematch` rows are upserted every run while the fixture is upcoming and never written after kickoff â€” historical pre-match stats must stay exactly as they were, unaffected by later matches.
- **Hot picks freeze at kickoff and settle exactly once:** `fixture_predictions` rows use the same `kickoff > NOW()` selection-freeze; the settle pass (canonical FT scores â†’ `result_goals`/`outcome`) is the only writer of those columns. The hit-rate scoreboard is honest by construction â€” never rewrite a settled or past-kickoff pick. AI adjudication is optional (no `GEMINI_API_KEY` = rules-only) and can only veto, never promote; API `/predictions` uses the fetch-once flag `fixtures.predictions_fetched_at`.
- **Fetch-once history:** `fixtures.history_fetched_at` guarantees each upcoming correlated fixture costs at most 3 backfill requests ever (2Ã— team last-N, 1Ã— full head-to-head). Only FINAL_STATUSES items are saved â€” headtohead also returns future meetings, which must not leak into the results refresh set.
- **Results are canonical:** the `results` action copies authoritative scores from final fixtures into linked matches (bookmaker-parsed scores are unreliable for upcoming games â€” BetPawa reports 0-0, Betika null).
- **Betika null fields:** `home_team_id, away_team_id, region_id, region_name, category_id, competition_id` are always null (not exposed by its API).
- Migrations are forward-only; never edit an applied migration.
